## API surface decision table

This guide compares the four major APIs available for AutoCAD‑based development at Chamber 19: AutoCAD .NET, AutoCAD JavaScript API, APS Design Automation and APS Viewer. Use it as your starting point when deciding which technology to use for a given task. Each API has strengths and limitations; choosing the right one up front avoids rework.

## Decision guide

If your tool runs inside AutoCAD and needs full access to entities, blocks, attributes and events, use the .NET API. It is the primary, in‑process API and exposes the complete object model. See the dotnet/ folder for patterns.

If your tool needs a web‑based UI hosted inside AutoCAD, consider the AutoCAD JavaScript API. It allows you to build HTML/JavaScript interfaces in a docked panel and call into the AutoCAD environment asynchronously. However, it cannot create or persist entities.

If you need to process drawings in the cloud or on a server without AutoCAD installed, use APS Design Automation. Submit a work item containing your plugin and DWG files; Autodesk’s service runs your code in a headless AutoCAD engine. Ideal for batch conversion, plotting, and metadata extraction.

If you need to view models in a browser and perform read‑only interactions, use APS Viewer. It is a WebGL viewer that loads 2D and 3D models in the browser with token‑based authentication. You can view properties but cannot modify DWG content.

In summary, choose the .NET API for in‑process plugins with full capabilities, the JavaScript API for lightweight UI extensions, APS Design Automation for headless batch operations, and APS Viewer for browser‑based viewing and property inspection.

## Transaction model (non-negotiable)

> Source: [dotnet/fundamentals/transaction-model.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/transaction-model.md)

This document explains how to use the transaction model in the AutoCAD .NET API and is the authoritative reference for Chamber 19 tools. Every developer writing C# code that interacts with the AutoCAD database must follow this pattern. The examples in this file are drawn from the official AutoCAD 2026 .NET training labs and show the correct way to open objects, add new entities, commit work and clean up.

## Overview

A transaction encapsulates a unit of work against the AutoCAD database. All reads and writes to the database occur within a transaction. When the transaction is committed, the changes persist; if the transaction is disposed without committing, all changes are rolled back.

The typical pattern is:

1. Start a transaction using `Database.TransactionManager.StartTransaction()`.
2. Open existing objects for read using `GetObject(id, OpenMode.ForRead)`.
3. When you need to modify an object, call `UpgradeOpen()` to promote it to write access. Only open `ForWrite` when needed to minimise locking.
4. Create new objects and append them to the appropriate table record, then call `AddNewlyCreatedDBObject()` so the transaction will manage their lifetime.
5. When all operations succeed, call `Commit()` to persist changes.
6. Use a `try/catch/finally` or `using` block to guarantee that `Dispose()` is called, rolling back uncommitted work automatically.

The following sections describe each aspect in detail.

## Correct transaction pattern

The following example (adapted from Lab 3 of the official training) creates a new circle in the current space. It follows the correct transaction pattern with a using block, opens objects ForRead and upgrades when necessary, registers newly created objects and commits only after all operations succeed.

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;
using Autodesk.AutoCAD.EditorInput;
using Autodesk.AutoCAD.Geometry;
using Autodesk.AutoCAD.Runtime;

public class TransactionExample
{
    [CommandMethod("CreateCircle")]
    public void CreateCircle()
    {
        Editor ed = Application.DocumentManager.MdiActiveDocument.Editor;
        // Prompt for centre and radius
        PromptPointResult ptRes = ed.GetPoint("\nPick centre point: ");
        if (ptRes.Status != PromptStatus.OK) return;
        PromptDoubleResult radRes = ed.GetDistance(new PromptDistanceOptions("\nPick radius:")
        {
            BasePoint = ptRes.Value,
            UseBasePoint = true
        });
        if (radRes.Status != PromptStatus.OK) return;

        Database db = ed.Document.Database;
        // Start a transaction in a using block so Dispose() is called automatically
        using (Transaction tr = db.TransactionManager.StartTransaction())
        {
            // Open the current space (ModelSpace or PaperSpace) for write
            BlockTableRecord currentSpace =
                (BlockTableRecord)tr.GetObject(db.CurrentSpaceId, OpenMode.ForWrite);

            // Create the new entity
            Circle circle = new Circle(ptRes.Value, Vector3d.ZAxis, radRes.Value);

            // Append the circle to the space and register it with the transaction
            currentSpace.AppendEntity(circle);
            tr.AddNewlyCreatedDBObject(circle, true);

            // Commit to persist changes
            tr.Commit();
        }
    }
}
```

## Why open ForRead first?

Always open existing objects ForRead, even if you intend to modify them later. Opening an object ForWrite places a write lock on it, which can block other commands and degrade performance. When you actually need to change a property, call UpgradeOpen() on the object to promote the access mode to ForWrite. This pattern keeps read operations fast and delays write locking until necessary.

```csharp
using (Transaction tr = db.TransactionManager.StartTransaction())
{
    BlockTable bt = tr.GetObject(db.BlockTableId, OpenMode.ForRead) as BlockTable;
    BlockTableRecord ms = tr.GetObject(bt[BlockTableRecord.ModelSpace], OpenMode.ForRead) as BlockTableRecord;

    foreach (ObjectId id in ms)
    {
        Entity ent = tr.GetObject(id, OpenMode.ForRead) as Entity;
        // Only upgrade when modifying
        if (ent is Circle)
        {
            ent.UpgradeOpen();     // promotes to ForWrite
            ent.ColorIndex = 1;    // change property safely
        }
    }
    tr.Commit();
}
```

## Registering new objects with AddNewlyCreatedDBObject

Whenever you create a new DBObject (such as Line, Circle, BlockReference, Xrecord, etc.), you must register it with the current transaction using Transaction.AddNewlyCreatedDBObject(newObject, true) after appending it to a container. Registration tells the transaction manager to manage the object's lifetime and include it in the commit or rollback. If you skip this step, the object may be created in memory but will not be persisted when you commit.

## Commit vs. rollback

Calling Commit() on a transaction makes all changes permanent. If you dispose of a transaction without calling Commit(), the API automatically rolls back the changes. This provides a safety net: if an exception occurs, the finally block disposes the transaction, and AutoCAD reverts the database to its previous state. Only call Commit() when all operations succeed.

## Common exceptions

- **`eNotOpenForWrite`** — thrown when you attempt to modify an object that was opened `ForRead`. Fix by calling `UpgradeOpen()` before setting any properties.
- **`eWasErased`** — thrown when you try to access an object that has been erased. Test `DBObject.IsErased` before calling `GetObject` or modify your logic to skip erased entities.
- **New object lost on commit** — occurs when you append an entity but forget to call `AddNewlyCreatedDBObject()`. Always register new objects with the transaction.

## Batch processing (headless)

For high‑volume operations on many drawings without launching AutoCAD's UI, use a headless pattern. Create a temporary Database object, read a DWG file, start a transaction, perform your work, commit, and finally save the file.

```csharp
using var db = new Database(false, true);
db.ReadDwgFile(inputPath, FileOpenMode.OpenForReadAndAllShare, false, null);
using var tr = db.TransactionManager.StartTransaction();
// Perform work on db here using the patterns above
tr.Commit();
// Save to a new file or overwrite the existing drawing
db.SaveAs(outputPath, DwgVersion.Current);
```

Use this pattern when batch processing hundreds of drawings or running in a server context. In‑process plugins should use HostApplicationServices.WorkingDatabase instead.

## Dynamic block name resolution

Dynamic block references often point at temporary anonymous definitions whose names start with *U. To obtain the real name of a dynamic block, use the DynamicBlockTableRecord property of a BlockReference. The dynamic block API returns the effective block table record for the block reference and allows you to determine the true name of the underlying block.

```csharp
BlockReference bref = …;
BlockTableRecord effective = (BlockTableRecord)tr.GetObject(
    bref.DynamicBlockTableRecord, OpenMode.ForRead);
string blockName = effective.Name; // real name, even for dynamic blocks
```

## Transaction failure modes

| Symptom | Cause | Fix |
| --- | --- | --- |
| eNotOpenForWrite when setting a property | Object was opened ForRead and not upgraded | Call entity.UpgradeOpen() before modifying properties |
| eWasErased when accessing an entity | The object has been erased in the current or a previous transaction | Test DBObject.IsErased and skip erased objects |
| New entity disappears after commit | AddNewlyCreatedDBObject() not called for the new object | After appending the entity, call tr.AddNewlyCreatedDBObject(newObj, true) |
| Changes lost when leaving a transaction | Transaction was disposed without calling Commit() | Call tr.Commit() only after all work succeeds; otherwise, the rollback will revert changes |
| Locking or performance issues when reading many objects | Objects opened ForWrite when only reading | Open objects with OpenMode.ForRead and call UpgradeOpen() only when necessary |
| ReadDwgFile fails due to sharing violation | The file is open in another process | Use FileOpenMode.OpenForReadAndAllShare to allow sharing or close other handles |
| Dynamic block shows a name like *U5 | The block reference points to an anonymous dynamic definition | Use BlockReference.DynamicBlockTableRecord to obtain the effective name |
| Error writing to database outside transaction | Attempting to modify database objects without a transaction | Always start a transaction and perform reads/writes within its scope |

## Why marshaling is required

`chamber-19-autocad-mcp` hosts HTTP endpoints inside `acad.exe`, so incoming MCP requests arrive on Kestrel worker threads rather than the thread that ran `IExtensionApplication.Initialize()`. The plugin captures the AutoCAD application thread during initialization, then uses `Application.Idle` to drain queued callbacks on that thread. The dispatcher exposes async helpers:

- `Task InvokeOnApplicationThreadAsync(Action)`
- `Task<T> InvokeOnApplicationThreadAsync<T>(Func<T>)`

Any tool that touches `Application.DocumentManager`, a `Database`, a `Transaction`, or entity state should go through this boundary. `ListBlocksTool` is a concrete example: the public async tool method dispatches to `ReadBlocks`, and all AutoCAD reads happen inside the application-thread callback. [chamber-19-autocad-mcp/README.md†L29-L33] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Tools/ListBlocksTool.cs†L35-L36] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Tools/ListBlocksTool.cs†L41-L46]

## Bounded queue pattern

The dispatcher keeps a bounded in-memory queue of pending callbacks. The default capacity is `32`, exposed as `AutoCadThreadDispatcher.DefaultQueueCapacity` and `QueueCapacity`. When the queue reaches capacity, new work is rejected instead of growing without bound. The tests lock in both the default value and the "queue is full" failure behavior. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Threading/AutoCadThreadDispatcher.cs†L24-L34] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Threading/AutoCadThreadDispatcher.cs†L48-L58] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp.Tests/AutoCadThreadDispatcherTests.cs†L24-L30] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp.Tests/AutoCadThreadDispatcherTests.cs†L50-L64]

This pattern matters because AutoCAD has a single UI/application thread. If client traffic outpaces the rate at which AutoCAD can drain work, unbounded buffering just moves the failure later and makes the UI less predictable. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Threading/AutoCadThreadDispatcher.cs†L10-L23]

## HTTP overload behavior

`chamber-19-autocad-mcp` adds HTTP middleware that checks queue depth before a request reaches the tool handler. When the queue depth is at or above capacity, the server returns:

- `429 Too Many Requests`
- `Retry-After: 1`
- JSON body with `error`, `error_description`, `queueDepth`, and `queueCapacity`

This gives clients a clean retry signal instead of letting requests pile up indefinitely behind a busy AutoCAD session. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/Backpressure.cs†L9-L24] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/Backpressure.cs†L35-L55]

## Tuning guidance

Treat `32` as a conservative default, not a universal truth.

- Keep the capacity low when tools do heavier drawing reads or any UI-sensitive work.
- Increase it only when tool calls are short, clients are bursty, and you have measured that AutoCAD drains the queue comfortably.
- Do not rely on a larger queue to fix slow tools; reduce work per request first.
- Clients should honor `429` and back off rather than busy-retrying.

Because the middleware is a best-effort pre-check, the dispatcher itself still needs to enforce the hard cap. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/Backpressure.cs†L14-L24] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp.Tests/AutoCadThreadDispatcherTests.cs†L50-L64]

## MSBuild setup

> Source: [dotnet/msbuild-setup.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/msbuild-setup.md)

This guide shows how to configure C# projects that target AutoCAD. The instructions here apply to in‑process ObjectARX managed plugins built with the .NET 8 SDK (AutoCAD 2026/2027) and older versions that still use .NET Framework 4.8 (AutoCAD 2024). Proper configuration prevents common issues such as missing commands, version mismatches and unexpected file copies.

## Target frameworks and project structure

AutoCAD 2026+ uses the .NET 8 platform on Windows. Older versions like AutoCAD 2024 require the classic .NET Framework 4.8. The following excerpt from an Autodesk Developer Blog sample shows how to multi‑target both frameworks:

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <AssemblyName>JigSample</AssemblyName>
    <OutputType>Library</OutputType>
    <RootNamespace>JigSample</RootNamespace>
    <TargetFramework>net8.0-windows</TargetFramework>
    <GenerateAssemblyInfo>False</GenerateAssemblyInfo>
    <Platforms>x64</Platforms>
    <Configurations>ACAD2024;ACAD2025</Configurations>
    <BaseOutputPath>bin</BaseOutputPath>
  </PropertyGroup>
  <PropertyGroup Condition="'$(Configuration)' == 'ACAD2025'">
    <TargetFramework>net8.0-windows</TargetFramework>
    <!-- Uncomment to enable WinForms or WPF -->
    <!--<UseWindowsForms>true</UseWindowsForms>
    <UseWPF>true</UseWPF>-->
    <AssemblySearchPaths>D:\ArxSdks\Arx2025\inc;$(AssemblySearchPaths)</AssemblySearchPaths>
  </PropertyGroup>
  <PropertyGroup Condition="'$(Configuration)' == 'ACAD2024'">
    <TargetFramework>net48</TargetFramework>
    <AssemblySearchPaths>D:\ArxSdks\Arx2024\inc;$(AssemblySearchPaths)</AssemblySearchPaths>
  </PropertyGroup>
  <ItemGroup Condition="'$(Configuration)' == 'ACAD2025'">
    <FrameworkReference Include="Microsoft.WindowsDesktop.App" />
  </ItemGroup>
  <ItemGroup>
    <Reference Include="AcDbMgd">
      <Private>False</Private>
    </Reference>
    <Reference Include="acmgd">
      <Private>False</Private>
    </Reference>
    <Reference Include="accoremgd">
      <Private>False</Private>
    </Reference>
  </ItemGroup>
</Project>
```

In this example the ACAD2025 configuration targets net8.0-windows while ACAD2024 targets net48. The FrameworkReference is added only for .NET 8 to bring in Windows desktop assemblies. Each Autodesk assembly (e.g. AcDbMgd, acmgd, accoremgd) has `<Private>False</Private>` so MSBuild does not copy the DLL into your output folder. Copying Autodesk assemblies into your plugin folder causes version conflicts and is a common source of crashes.

## Directory.Build.props

Rather than hard‑coding installation paths in every project, define properties in a shared Directory.Build.props file. This file sits at the repository root and is imported automatically by MSBuild. For example:

```xml
<Project>
  <PropertyGroup>
    <!-- Path to the AutoCAD installation.  On developer machines this might be
         C:\Program Files\Autodesk\AutoCAD 2026. -->
    <AcadDir>C:\\Program Files\\Autodesk\\AutoCAD 2026</AcadDir>
    <!-- ObjectARX SDK path if you need header files or unmanaged libraries. -->
    <ArxSdk>C:\\ObjectARX 2026</ArxSdk>
    <!-- Derived search paths for managed assemblies. -->
    <ArxMgdPath>$(AcadDir)</ArxMgdPath>
    <AssemblySearchPaths>$(ArxSdk)\inc;$(AssemblySearchPaths)</AssemblySearchPaths>
  </PropertyGroup>
</Project>
```

Your individual `.csproj` can then reference assemblies relative to these variables. The AutoCAD Net Wizards repository demonstrates this pattern: each reference sets HintPath to $(ArxMgdPath)\acdbmgd.dll and marks `<Private>False</Private>`. When you upgrade to a new AutoCAD version you only need to change AcadDir in one place.

## Choosing NETLOAD vs demand‑load

There are two ways to load managed plugins into AutoCAD:

- **NETLOAD** — manually load the DLL from within AutoCAD. This is the simplest option during development. Start AutoCAD, run the `NETLOAD` command and browse to your compiled DLL. Commands exposed via `[CommandMethod]` become available immediately.
- **Demand‑load (registry or package)** — automatically load your plugin when AutoCAD starts or when certain commands are invoked. This requires writing registry entries (for legacy .NET Framework plugins) or providing a `PackageContents.xml` file for modern versions. Demand‑load is more complex but necessary for production deployments.

Development teams often use NETLOAD during debugging and configure demand‑load through the App Loader mechanism for shipping. Never copy Autodesk assemblies into your package – rely on the installed versions referenced via HintPath with `<Private>`False</Private>.

## Debugging

To debug your plugin, set up a launch profile that starts AutoCAD with your DLL pre‑loaded. The AutoCAD Net Wizards sample defines multiple profiles in launchSettings.json; one of them launches acad.exe directly. In Visual Studio you can select this profile and press F5. Alternatively, build your project, run AutoCAD manually, load the DLL with NETLOAD and then attach to the acad.exe process from Visual Studio.

## Minimal helper shape

The current pattern is intentionally simple:

```csharp
internal static class AutoCadCompatibility
{
    public const int MinimumSupportedMajor = 24;

    public static bool IsSupported(Version? version) =>
        version is not null && version.Major >= MinimumSupportedMajor;

    public static string Describe(Version? version)
    {
        if (version is null)
            return $"AutoCAD version: unknown (floor R{MinimumSupportedMajor}.0).";

        if (version.Major >= MinimumSupportedMajor)
            return $"AutoCAD R{version} (supported; floor R{MinimumSupportedMajor}.0).";

        return $"AutoCAD R{version} is below the supported floor (R{MinimumSupportedMajor}.0+).";
    }
}
```

This gives the host two useful behaviors:

- `IsSupported` — a pure predicate suitable for gating startup
- `Describe` — a human-readable status message for logs, diagnostics, or commands such as `MCPSTATUS`

## MCP discovery and connection

> Source: [dotnet/fundamentals/mcp-discovery.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/mcp-discovery.md)

This guide documents the discovery contract used by `chamber-19-autocad-mcp` when third-party MCP clients connect to AutoCAD over HTTP. The current plugin writes one discovery file per AutoCAD process rather than sharing a single `port.txt`, which avoids collisions when multiple AutoCAD sessions are open at once. [chamber-19-autocad-mcp/README.md†L3-L5] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/PortFile.cs†L7-L18]

## Discovery file path

The canonical location is:

```text
%LOCALAPPDATA%\Chamber19\autocad-mcp\port.<pid>.txt
```

The `<pid>` suffix makes the layout multi-instance safe. Two AutoCAD sessions can publish discovery metadata at the same time without racing on the same file, and clients are expected to glob `port.*.txt` under the directory to discover all running servers on the machine. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/PortFile.cs†L7-L18] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/PortFile.cs†L25-L30]

## JSON payload

Each file contains a compact JSON object with four fields:

```json
{
  "url": "http://127.0.0.1:5001/",
  "token": "token-abc",
  "pid": 1234,
  "started": "2026-05-08T12:00:00.0000000+00:00"
}
```

- `url` — HTTP base URL for the MCP server
- `token` — bearer token clients must send
- `pid` — AutoCAD process id that owns the server
- `started` — process start time in round-trip (`O`) format

The implementation and tests lock in both the field set and the field order. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/PortFile.cs†L36-L52] [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp.Tests/PortFileTests.cs†L37-L57]

## Client discovery flow

Use this client-side pattern:

1. Enumerate `%LOCALAPPDATA%\Chamber19\autocad-mcp\port.*.txt`.
2. Parse each JSON payload.
3. Reject malformed files.
4. Reject stale files whose `pid` is no longer running.
5. When possible, compare the live process start time to `started` and reject mismatches.
6. If multiple valid servers remain, either ask the user which AutoCAD session to use or apply an explicit policy such as "newest `started` wins".

The discovery file is a hint, not a lease. Cleanup is best-effort, so clients should validate the live process before trusting a file. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/PortFile.cs†L55-L68]

## Stale files after crashes

Normal shutdown deletes the per-process file, but forced termination can leave stale files behind. The next clean launch overwrites its own file, but it cannot remove stale files from dead processes automatically. Client code should therefore treat an unreadable file, a dead `pid`, or a `started` mismatch as a stale discovery record and ignore it. [chamber-19-autocad-mcp/dotnet/Chamber19.AutoCad.Mcp/Hosting/PortFile.cs†L49-L68]

## Editor input patterns

> Source: [dotnet/fundamentals/editor-input.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/editor-input.md)

This guide covers the standard input mechanisms used by the AutoCAD .NET API. Most commands interact with users through the Editor object, which exposes methods for prompting points, distances, numbers, keywords and strings. Understanding how to configure prompt options and interpret results is essential for building intuitive commands. The examples below draw from the official training labs and demonstrate common patterns.

## Keyword prompts

When you need the user to choose from a set of options, use PromptKeywordOptions and Editor.GetKeywords(). You supply a message and a list of acceptable keywords. The return value is a PromptResult whose Status indicates success and whose StringResult contains the selected keyword.

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.EditorInput;
using Autodesk.AutoCAD.Runtime;

[CommandMethod("ChooseShape")]
public static void ChooseShape()
{
    var ed = Application.DocumentManager.MdiActiveDocument.Editor;
    var opts = new PromptKeywordOptions("Pick a shape [Circle/Block] : ", "Circle Block");
    PromptResult res = ed.GetKeywords(opts);
    if (res.Status != PromptStatus.OK)
        return;
    switch (res.StringResult)
    {
        case "Circle":
            ed.WriteMessage("\nYou chose a circle.");
            break;
        case "Block":
            ed.WriteMessage("\nYou chose a block.");
            break;
    }
}
```

`PromptKeywordOptions` has properties to control behaviour, such as AllowNone (allow pressing Enter without choosing) and Keywords collection for adding more keywords. Always test PromptResult.Status before using the result to handle user cancellations gracefully.

## Point input

Use PromptPointOptions and Editor.GetPoint() when you need a 3D point. You can specify a base point, enforce use of the base point, or restrict to the current UCS. The returned PromptPointResult exposes the selected point via its Value property and a Status to indicate success.

```csharp
PromptPointOptions pointOpts = new PromptPointOptions("Pick center point: ");
PromptPointResult pointRes = ed.GetPoint(pointOpts);
if (pointRes.Status != PromptStatus.OK)
    return;
Point3d center = pointRes.Value;
```

## Distance and numeric input

For distances relative to a base point, use PromptDistanceOptions and Editor.GetDistance(). Set BasePoint to the fixed point and UseBasePoint to true so AutoCAD knows to measure from it. You can also set DefaultValue and AllowNegative for other numeric prompts.

```csharp
PromptDistanceOptions distOpts = new PromptDistanceOptions("Pick radius: ");
distOpts.BasePoint = center;
distOpts.UseBasePoint = true;
PromptDoubleResult distRes = ed.GetDistance(distOpts);
if (distRes.Status != PromptStatus.OK)
    return;
double radius = distRes.Value;
```

For arbitrary numeric input, use PromptDoubleOptions or PromptIntegerOptions with Editor.GetDouble() or Editor.GetInteger(). These options allow you to set limits (LowerLimit, UpperLimit) and default values.

## String input

PromptStringOptions prompts the user for text. You can set AllowSpaces to permit spaces in the input. Use Editor.GetString() to obtain a PromptResult containing the entered text.

```csharp
PromptStringOptions strOpts = new PromptStringOptions("Enter block name: ");
strOpts.AllowSpaces = false;
PromptResult strRes = ed.GetString(strOpts);
if (strRes.Status == PromptStatus.OK)
{
    string blockName = strRes.StringResult;
    // use the block name
}
```

## Handling PromptStatus

All prompt methods return a result object with a Status property. The status can be:

| Status | Meaning |
| --- | --- |
| OK | User provided a valid input. Proceed using the value. |
| Cancel | User pressed Esc or cancelled the command. You should abort your command. |
| Error | Invalid input or other failure. Display an error message and abort. |

Always check Status before using the result to avoid exceptions or unintended behaviour.

## Combining prompts into workflows

Complex commands often chain multiple prompts. The training labs provide a complete example that asks for a shape type, a point and a radius to create a circle. The pattern is:

1. Prompt for a keyword to decide the operation.
2. If the user chooses a circle, prompt for a point and a distance.
3. If the user chooses a block, prompt for a string (block name).

When chaining prompts, return early on any cancellation. Only proceed to create database objects after all inputs are valid. See the examples/lab2/ folder for the full code.

## Editor input summary

Use the Editor class to interact with users through prompt methods. Configure the appropriate options object (PromptKeywordOptions, PromptPointOptions, PromptDistanceOptions, etc.), call the corresponding Get… method on the editor, and check the returned Status before using the result. Chaining prompts allows you to collect multiple inputs for complex operations.

## Selection sets

> Source: [dotnet/fundamentals/selection-sets.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/selection-sets.md)

This file explains how to build and use selection sets in the AutoCAD .NET API. Selection sets are collections of objects that meet certain criteria, defined by a set of DXF group codes and values. By combining TypedValue filters and editor methods like SelectAll and SelectCrossingWindow you can quickly find and iterate entities across a drawing. You should read this guide if you need to query entities by type, layer, name or other properties, or if you need to window‑select objects programmatically.

## Constructing a SelectionFilter

A SelectionFilter takes an array of TypedValue objects. Each TypedValue pairs a DxfCode (the filter type) with a value. When you pass the filter to the editor’s selection methods, AutoCAD returns only those objects that satisfy all filter conditions. Common filter codes include:

| Purpose | DxfCode | Example value |
| --- | --- | --- |
| Entity type | Start | "LINE", "CIRCLE", "INSERT" |
| Block name | BlockName | "TITLEBLOCK", "RELAY_*" (wild‑cards allowed) |
| Layer name | LayerName | "P-CTL*" to match discipline layers |
| Color | Color | 1 (red), 256 (ByLayer) |
| Line type | LinetypeName | "CENTER", "HIDDEN" |

Wild‑card characters such as `*` (any sequence) and `?` (any single character) can be used in string values. Autodesk’s documentation lists these wild‑cards and their meanings. For example, to select all insertions of relay blocks whose names start with `RELAY_` you can filter on `DxfCode.BlockName` with value `RELAY_*`.

## Selecting all objects that match a filter

To select every object in a drawing that satisfies your filter, call Editor.SelectAll(filter). The following example selects all circles on a specific layer and iterates through them:

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;
using Autodesk.AutoCAD.EditorInput;

[CommandMethod("CountCircles")]
public static void CountCircles()
{
    var ed = Application.DocumentManager.MdiActiveDocument.Editor;
    var db = Application.DocumentManager.MdiActiveDocument.Database;
    // filter: circles on the P-CTL layer
    TypedValue[] tvs = new[]
    {
        new TypedValue((int)DxfCode.Start, "CIRCLE"),
        new TypedValue((int)DxfCode.LayerName, "P-CTL*")
    };
    var filter = new SelectionFilter(tvs);
    PromptSelectionResult res = ed.SelectAll(filter);
    if (res.Status != PromptStatus.OK)
    {
        ed.WriteMessage("\nNo circles found.");
        return;
    }
    using (var tr = db.TransactionManager.StartTransaction())
    {
        int count = 0;
        foreach (SelectedObject sel in res.Value)
        {
            var ent = (Entity)tr.GetObject(sel.ObjectId, OpenMode.ForRead);
            // process entity – here we just count it
            count++;
        }
        ed.WriteMessage($"\nFound {count} circle(s).");
        tr.Commit();
    }
}
```

The `SelectAll` method returns a PromptSelectionResult. Always check its Status property before accessing the SelectionSet. Iterate through res.Value to process each SelectedObject. Use a transaction to open objects for read or write as needed.

## Selecting within a rectangular window

To select entities within a defined rectangle, use SelectCrossingWindow or SelectWindow. SelectCrossingWindow selects objects that intersect or lie inside the window; SelectWindow selects only objects entirely inside. You supply two corner points:

```csharp
Point3d p1 = new Point3d(0, 0, 0);
Point3d p2 = new Point3d(100, 50, 0);
// optional filter: only lines
TypedValue[] tvs = { new TypedValue((int)DxfCode.Start, "LINE") };
SelectionFilter filter = new SelectionFilter(tvs);
PromptSelectionResult res = ed.SelectCrossingWindow(p1, p2, filter);
if (res.Status == PromptStatus.OK)
{
    foreach (SelectedObject so in res.Value)
    {
        // open and process each line
    }
}
```

You can combine spatial selection with property filters to narrow results. If you omit the filter argument, all objects within the window will be returned.

## Iterating a SelectionSet

A SelectionSet is an unordered collection of selected objects. Each element is a SelectedObject which contains the ObjectId of the entity and the method of selection. To work with the actual entities, open them within a transaction:

```csharp
PromptSelectionResult res = ed.GetSelection();
if (res.Status == PromptStatus.OK)
{
    using (var tr = db.TransactionManager.StartTransaction())
    {
        foreach (SelectedObject sel in res.Value)
        {
            var ent = (Entity)tr.GetObject(sel.ObjectId, OpenMode.ForRead);
            ed.WriteMessage($"\nEntity type: {ent.GetType().Name}, on layer {ent.Layer}");
        }
        tr.Commit();
    }
}
```

The `GetSelection()` method prompts the user to select objects using the standard AutoCAD selection tools (clicking, windowing, etc.). Use this when interactive selection is acceptable.

## Using wild‑cards and advanced filters

The SelectionFilter understands wild‑card patterns defined in AutoCAD’s selection filtering rules. * matches any sequence of characters and ? matches a single character. You can negate patterns with ~ and specify lists or ranges within square brackets. For example, to select all blocks whose names start with RLY or SW:

```csharp
TypedValue[] tvs = new[]
{
    new TypedValue((int)DxfCode.Start, "INSERT"),
    new TypedValue((int)DxfCode.BlockName, "RLY*,SW*")
};
```

For complex conditions that cannot be expressed by DXF codes—such as selecting all entities with a specific property value or dynamic block attribute—you may need to select a broader set and then filter in C#. After obtaining the SelectionSet, iterate through each entity, upgrade it to ForRead and apply your custom logic.

## Selection sets summary

Selection filters and sets provide a powerful way to query drawings programmatically. Use TypedValue arrays and appropriate DxfCodes to define criteria, then call SelectAll, SelectWindow or GetSelection depending on whether you want a programmatic or interactive selection. Always open entities within a transaction before reading or modifying them.

## Layer patterns

> Source: [dotnet/fundamentals/layers.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/layers.md)

Layers control the visibility and appearance of entities in AutoCAD drawings. Each drawing contains a LayerTable which holds LayerTableRecord objects. This document explains how to list layers, create new layers, modify layer properties and assign entities to layers using the AutoCAD .NET API. Read this guide when you need to manage layers programmatically or ensure that entities are drawn on the correct layer.

## Listing and retrieving layers

To iterate over all layers in a drawing, open the LayerTable for read and then open each LayerTableRecord for read:

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;
using Autodesk.AutoCAD.Runtime;

[CommandMethod("ListLayers")]
public static void ListLayers()
{
    var db = Application.DocumentManager.MdiActiveDocument.Database;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        LayerTable lt = (LayerTable)tr.GetObject(db.LayerTableId, OpenMode.ForRead);
        foreach (ObjectId id in lt)
        {
            LayerTableRecord ltr = (LayerTableRecord)tr.GetObject(id, OpenMode.ForRead);
            Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nLayer: {ltr.Name}, Color: {ltr.Color}");
        }
        tr.Commit();
    }
}
```

You can test whether a layer exists by calling lt.Has(layerName). Always open the LayerTable ForRead before reading and upgrade it to ForWrite before adding or deleting layers.

## Creating a new layer

Creating a layer involves instantiating a LayerTableRecord, setting its properties, adding it to the LayerTable and registering it with the transaction. The following command creates a layer named P-CTL, sets its color to yellow (AutoCAD color index 2), assigns the Continuous linetype and ensures the linetype is loaded into the drawing:

```csharp
[CommandMethod("CreateLayer")]
public static void CreateLayer()
{
    var doc = Application.DocumentManager.MdiActiveDocument;
    var db  = doc.Database;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        LayerTable lt = (LayerTable)tr.GetObject(db.LayerTableId, OpenMode.ForRead);
        const string layerName = "P-CTL";
        if (!lt.Has(layerName))
        {
            lt.UpgradeOpen();
            var newLayer = new LayerTableRecord
            {
                Name = layerName,
                Color = Autodesk.AutoCAD.Colors.Color.FromColorIndex(Autodesk.AutoCAD.Colors.ColorMethod.ByAci, 2),
                LinetypeObjectId = db.ContinuousLinetype
            };
            lt.Add(newLayer);
            tr.AddNewlyCreatedDBObject(newLayer, true);
        }
        tr.Commit();
    }
}
```

If the linetype you need is not already loaded, call Database.LoadLineTypeFile() before setting the LinetypeObjectId.

## Changing layer properties

To modify a layer, open its LayerTableRecord ForWrite, change its properties and commit the transaction. Example: rename a layer and set it to be frozen in the current viewport:

```csharp
[CommandMethod("ModifyLayer")]
public static void ModifyLayer()
{
    var db = Application.DocumentManager.MdiActiveDocument.Database;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        LayerTable lt = (LayerTable)tr.GetObject(db.LayerTableId, OpenMode.ForRead);
        if (lt.Has("OLD_NAME"))
        {
            LayerTableRecord ltr = (LayerTableRecord)tr.GetObject(lt["OLD_NAME"], OpenMode.ForWrite);
            ltr.Name = "NEW_NAME";
            ltr.IsFrozen = true;
        }
        tr.Commit();
    }
}
```

Other properties you can set include IsOff (turn layer off), IsLocked (prevent modifications), IsPlotting (control whether the layer plots) and Color/LinetypeObjectId to change appearance.

## Assigning entities to layers

An entity’s Layer property holds the name of its layer. To move an entity to a different layer, open it ForWrite and set the Layer property to the target layer name:

```csharp
[CommandMethod("MoveToLayer")]
public static void MoveToLayer()
{
    var ed = Application.DocumentManager.MdiActiveDocument.Editor;
    var db = Application.DocumentManager.MdiActiveDocument.Database;
    // ask the user for an entity and the target layer
    PromptEntityResult per = ed.GetEntity("\nSelect entity: ");
    if (per.Status != PromptStatus.OK) return;
    PromptStringOptions pso = new PromptStringOptions("\nEnter target layer name: ");
    pso.AllowSpaces = false;
    PromptResult pr = ed.GetString(pso);
    if (pr.Status != PromptStatus.OK) return;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        var ent = (Entity)tr.GetObject(per.ObjectId, OpenMode.ForWrite);
        ent.Layer = pr.StringResult;
        tr.Commit();
    }
}
```

Ensure the target layer exists before assigning entities. Creating layers dynamically at runtime can simplify workflows but may violate drawing standards if not controlled.

## Deleting layers

Layers can only be erased if they contain no entities, are not the current layer and are not required (e.g. layer 0, Defpoints or layers referenced by blocks). To delete a layer, open it ForWrite and call LayerTableRecord.Erase():

```csharp
[CommandMethod("DeleteLayer")]
public static void DeleteLayer()
{
    var db = Application.DocumentManager.MdiActiveDocument.Database;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        LayerTable lt = (LayerTable)tr.GetObject(db.LayerTableId, OpenMode.ForRead);
        if (lt.Has("UNUSED_LAYER"))
        {
            var ltr = (LayerTableRecord)tr.GetObject(lt["UNUSED_LAYER"], OpenMode.ForWrite);
            if (!ltr.IsUsed && !ltr.IsErased && !ltr.IsDependent)
            {
                ltr.Erase();
            }
        }
        tr.Commit();
    }
}
```

If the layer contains entities, you must move or delete them before erasing the layer. Deleting required layers will raise exceptions.

## Layer patterns summary

Layer management in AutoCAD .NET revolves around the LayerTable and LayerTableRecord. List layers by iterating the table, create new layers by adding LayerTableRecords and register them with the transaction, modify layer properties by opening records ForWrite, and assign entities to layers by setting the Layer property. Remember to respect drawing standards and ensure layers exist before using them.

## Block attributes

> Source: [dotnet/entities/attributes.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/entities/attributes.md)

This document explains how to work with block attributes in AutoCAD's .NET API. It is intended for developers building Chamber 19 tools that need to read or modify title block information and other attribute‐driven data. Understanding the difference between definitions and instances, how to locate block references and how to synchronise attributes are essential skills for productive automation.

## AttributeDefinition vs AttributeReference

In a block definition, an AttributeDefinition specifies the attribute's tag, prompt and default value. It exists only in the block definition (BlockTableRecord) and never holds actual values. When a block is inserted into a drawing, AutoCAD creates an AttributeReference for each AttributeDefinition. The AttributeReference belongs to the BlockReference and stores the current value of the attribute. Always read and write the AttributeReference, not the AttributeDefinition.

## Reading all attributes from a block reference

Use the AttributeCollection property of a BlockReference to iterate over its attributes. Open each attribute for read and inspect its Tag and TextString. The following example prompts the user to select a block reference and then prints each tag/value pair to the command line.

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;
using Autodesk.AutoCAD.EditorInput;
using Autodesk.AutoCAD.Runtime;

public void ListAttributes()
{
    Editor ed = Application.DocumentManager.MdiActiveDocument.Editor;
    Database db = ed.Document.Database;

    using (Transaction tr = db.TransactionManager.StartTransaction())
    {
        // Prompt for a block reference
        PromptEntityOptions peo = new PromptEntityOptions("\nSelect block: ");
        peo.SetRejectMessage("\nOnly block references");
        peo.AddAllowedClass(typeof(BlockReference), false);
        PromptEntityResult per = ed.GetEntity(peo);
        if (per.Status != PromptStatus.OK) return;

        BlockReference bref = (BlockReference)tr.GetObject(per.ObjectId, OpenMode.ForRead);
        foreach (ObjectId attId in bref.AttributeCollection)
        {
            AttributeReference att = (AttributeReference)tr.GetObject(attId, OpenMode.ForRead);
            ed.WriteMessage($"\n{att.Tag} = {att.TextString}");
        }
    }
}
```

## Writing an attribute value

To modify an attribute, open it ForRead, call UpgradeOpen() and then set its TextString. Calling RecordGraphicsModified() on the block reference forces a visual update. The following example finds the DRAWING_NO attribute and sets it to a new value.

```csharp
using (Transaction tr = db.TransactionManager.StartTransaction())
{
    BlockReference bref = (BlockReference)tr.GetObject(brefId, OpenMode.ForWrite);
    foreach (ObjectId attId in bref.AttributeCollection)
    {
        AttributeReference attRef = (AttributeReference)tr.GetObject(attId, OpenMode.ForRead);
        if (attRef.Tag.Equals("DRAWING_NO", StringComparison.OrdinalIgnoreCase))
        {
            attRef.UpgradeOpen();
            attRef.TextString = "D-1234";
        }
    }
    // Inform AutoCAD that the block reference needs to be redrawn
    bref.RecordGraphicsModified(true);
    tr.Commit();
}
```

## Finding all instances of a named block

To locate every instance of a block in model space, iterate the BlockTableRecord for model space, test each entity for BlockReference, and compare the block name. For dynamic blocks, obtain the effective block table record via DynamicBlockTableRecord because the reference may point to an anonymous *U block.

```csharp
List<ObjectId> FindBlocks(string blockName, Transaction tr, Database db)
{
    List<ObjectId> results = new();
    BlockTable bt = (BlockTable)tr.GetObject(db.BlockTableId, OpenMode.ForRead);
    BlockTableRecord ms = (BlockTableRecord)tr.GetObject(bt[BlockTableRecord.ModelSpace], OpenMode.ForRead);

    foreach (ObjectId id in ms)
    {
        if (id.ObjectClass.DxfName == "INSERT")
        {
            BlockReference bref = (BlockReference)tr.GetObject(id, OpenMode.ForRead);
            // Get effective name for dynamic blocks
            BlockTableRecord effective = (BlockTableRecord)tr.GetObject(bref.DynamicBlockTableRecord, OpenMode.ForRead);
            string name = effective.Name;
            if (name.Equals(blockName, StringComparison.OrdinalIgnoreCase))
            {
                results.Add(id);
            }
        }
    }
    return results;
}
```

## Counting block references by definition

If you need counts for every named block definition rather than a list of matching inserts, use a two-pass walk:

1. Walk the `BlockTable` and collect candidate definitions.
2. Skip `BlockTableRecord` entries where `IsAnonymous` or `IsLayout` is `true`.
3. Walk every layout `BlockTableRecord` (model space and paper spaces).
4. For each `BlockReference`, resolve `DynamicBlockTableRecord` and increment the count for the effective definition name.

This avoids two common errors:

- counting anonymous `*U` dynamic variants as separate blocks
- counting block definitions or layout containers as references

```csharp
Dictionary<string, int> CountBlocks(Transaction tr, Database db)
{
    Dictionary<string, int> counts = new(StringComparer.OrdinalIgnoreCase);
    BlockTable bt = (BlockTable)tr.GetObject(db.BlockTableId, OpenMode.ForRead);

    // Pass 1: collect user block definitions.
    foreach (ObjectId btrId in bt)
    {
        BlockTableRecord btr = (BlockTableRecord)tr.GetObject(btrId, OpenMode.ForRead);
        if (btr.IsAnonymous || btr.IsLayout)
            continue;

        counts[btr.Name] = 0;
    }

    // Pass 2: walk layout BTRs and count inserts against the effective definition.
    foreach (ObjectId btrId in bt)
    {
        BlockTableRecord layoutBtr = (BlockTableRecord)tr.GetObject(btrId, OpenMode.ForRead);
        if (!layoutBtr.IsLayout)
            continue;

        foreach (ObjectId entityId in layoutBtr)
        {
            DBObject obj = tr.GetObject(entityId, OpenMode.ForRead);
            if (obj is not BlockReference bref)
                continue;

            BlockTableRecord effective =
                (BlockTableRecord)tr.GetObject(bref.DynamicBlockTableRecord, OpenMode.ForRead);

            if (counts.ContainsKey(effective.Name))
                counts[effective.Name]++;
        }
    }

    return counts;
}
```

This is the pattern used by `chamber19_list_blocks`: pass 1 collects non-anonymous, non-layout definitions, and pass 2 resolves each insert through `DynamicBlockTableRecord` so customized dynamic variants count against the original block definition. ## Title block automation pattern

Substation and protection drawings use a standard title block containing fields such as project number, drawing number, revision, date and engineer initials. To automate these fields:

1. Identify the title block by name (for example, `ANSI_A_TITLE_BLOCK`) using the `FindBlocks` function above.
2. For each `BlockReference` returned, iterate its `AttributeCollection`.
3. Use the attribute tag to find the field you want. Common tags include `PROJ_NO`, `DWG_NO`, `REV`, `DATE` and `ENGINEER`.
4. Read or update `TextString` as needed.
5. After modifying values, call `RecordGraphicsModified(true)` on the block reference so the change appears.

This pattern allows batch extraction of title block metadata (drawing register generation) and automated updates to revision or date fields.

## Accessing attributes by tag name

Do not rely on attribute order; attribute indices change when a block definition is modified. Instead, search by tag name. The Tag property is case‑insensitive but unique within a block definition. Use a helper function:

```csharp
AttributeReference? FindAttribute(BlockReference bref, Transaction tr, string tag)
{
    foreach (ObjectId attId in bref.AttributeCollection)
    {
        AttributeReference att = (AttributeReference)tr.GetObject(attId, OpenMode.ForRead);
        if (att.Tag.Equals(tag, StringComparison.OrdinalIgnoreCase))
            return att;
    }
    return null;
}
```

## Synchronising attributes

When a block definition is updated to add or remove attributes, existing block references may not automatically acquire the new attributes. To synchronise:

1. Open the block definition (`BlockTableRecord`) for write.
2. Call routines to sync attributes. AutoCAD does not provide a direct managed method in .NET 8, but typical patterns include erasing and reinserting the block reference, or using `RecordGraphicsModified()` to force regeneration. An alternative is to call the native `InsertBlockReference` API again.
3. For complex cases, consider recreating the `BlockReference` using the updated definition so that new `AttributeReference` objects are created automatically.

## Block attributes failure modes

| Symptom | Cause | Fix |
| --- | --- | --- |
| Changing TextString causes eNotOpenForWrite | Attribute was opened ForRead | Call UpgradeOpen() before modifying the attribute |
| Attributes missing from a block reference | Block definition changed after insertion | Synchronise attributes by reinserting the block or using attribute sync routines |
| Wrong attribute updated | Code iterated attributes by index instead of tag | Always find attributes by tag name rather than index |
| Dynamic block has anonymous name | Dynamic blocks use a temporary definition name starting with *U | Use DynamicBlockTableRecord to get the effective name |
| Dynamic block counts are split across `*U` records | Code counted `BlockReference.BlockTableRecord` or anonymous definitions directly | First collect non-anonymous, non-layout definitions, then count each insert against `DynamicBlockTableRecord` |
| Attribute values not visible after change | The block reference graphics were not updated | Call RecordGraphicsModified(true) on the BlockReference |
| eWasErased when reading attributes | The block reference or attribute was erased | Check IsErased on the BlockReference and skip erased objects |

This guidance should be followed whenever reading or writing block attributes. Accurate title block automation is critical for Chamber 19 engineering tools.

## XData

> Source: [dotnet/entities/xdata.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/entities/xdata.md)

Extended entity data, or XData, lets you attach arbitrary information to AutoCAD objects. XData is a lightweight alternative to extension dictionaries: it stores a list of TypedValues on the entity itself and is visible to all applications that know the application name. This file explains how to register your application name, read and write XData, and lists common pitfalls. Read this guide when you need to persist custom data with drawing entities beyond the built‑in properties and attributes.

## Registering your application name

Every XData record must begin with a DxfCode.AppName TypedValue whose value is your application name. Before you can attach XData you must register this name in the drawing’s RegAppTable. If you skip this step, AutoCAD throws an exception when you assign XData. The following helper registers an application name if it does not already exist:

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;

private static void EnsureRegApp(Database db, Transaction tr, string appName)
{
    RegAppTable rat = (RegAppTable)tr.GetObject(db.RegAppTableId, OpenMode.ForRead);
    if (!rat.Has(appName))
    {
        rat.UpgradeOpen();
        RegAppTableRecord reg = new RegAppTableRecord { Name = appName };
        rat.Add(reg);
        tr.AddNewlyCreatedDBObject(reg, true);
    }
}
```

Call this routine before writing XData to ensure the application name exists in the drawing.

## Writing XData to an entity

To attach XData, open the target entity for write, build a ResultBuffer with your TypedValues, and assign it to the entity’s XData property. The first element of the buffer must be a DxfCode.AppName with your registered name. Additional values can use other DxfCode types such as Real, Integer, Text, LayerName, etc. Example:

```csharp
using Autodesk.AutoCAD.EditorInput;
[CommandMethod("AddXData")]
public static void AddXData()
{
    var doc = Application.DocumentManager.MdiActiveDocument;
    var ed  = doc.Editor;
    var db  = doc.Database;
    // ask the user to pick an entity
    PromptEntityResult res = ed.GetEntity("\nSelect entity to add XData to: ");
    if (res.Status != PromptStatus.OK) return;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        const string appName = "MY_APP";
        EnsureRegApp(db, tr, appName);
        var ent = (Entity)tr.GetObject(res.ObjectId, OpenMode.ForWrite);
        // build the XData buffer: AppName must be first
        var rb = new ResultBuffer(
            new TypedValue((int)DxfCode.AppName, appName),
            new TypedValue((int)DxfCode.Real, 42.0),
            new TypedValue((int)DxfCode.Text, "Hello")
        );
        ent.XData = rb;
        tr.Commit();
    }
}
```

After executing this command, the selected entity will carry XData with your application name. You can store multiple values in a single ResultBuffer; choose DxfCode values appropriate to the data types you need.

## Reading XData

To retrieve XData from an entity, open it for read and access its XData property. The property returns a ResultBuffer or null if no data exists. Iterate the buffer’s TypedValues to inspect your data:

```csharp
[CommandMethod("ListXData")]
public static void ListXData()
{
    var ed = Application.DocumentManager.MdiActiveDocument.Editor;
    PromptEntityResult res = ed.GetEntity("\nSelect entity: ");
    if (res.Status != PromptStatus.OK) return;
    var db = Application.DocumentManager.MdiActiveDocument.Database;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        var ent = (Entity)tr.GetObject(res.ObjectId, OpenMode.ForRead);
        ResultBuffer rb = ent.XData;
        if (rb == null)
        {
            ed.WriteMessage("\nNo XData found.");
            return;
        }
        foreach (TypedValue tv in rb)
        {
            ed.WriteMessage($"\n{tv.TypeCode}: {tv.Value}");
        }
        tr.Commit();
    }
}
```

The first `TypedValue` will have TypeCode DxfCode.AppName. You can check it to ensure the XData belongs to your application before interpreting subsequent values.

## Using XData vs extension dictionaries

XData is limited in size and structure. All data must fit within the maximum allowed record length and must follow the sequence of TypedValues. If your data is complex, structured or large, consider using an extension dictionary instead, as shown in the training labs. Extension dictionaries store keyed XRecords and can hold arbitrary ResultBuffers. XData is best for small, simple data sets that need to be read by multiple applications.

## Failure modes and troubleshooting

| Symptom | Cause | Fix |
| --- | --- | --- |
| eInvalidInput when assigning XData | DxfCode.AppName missing or application name not registered | Ensure the first TypedValue in the ResultBuffer is DxfCode.AppName and call EnsureRegApp() before writing |
| XData appears truncated or missing values | Total size exceeds the XData record limit | Use fewer values or move data to an extension dictionary |
| XData property returns null | No XData attached to the entity | Attach XData first or check extension dictionaries |
| Reading returns unexpected TypeCode values | Different application wrote XData to the entity | Always verify the app name before parsing and ignore records that aren’t yours |
| Cannot store complex objects in XData | XData only supports basic types (String, Real, Int16, Int32, Handle, etc.) | Serialize complex structures into strings or use an extension dictionary |

## XData summary

XData provides a simple way to attach small amounts of custom data to AutoCAD entities. Always register your application name in the RegAppTable, build a ResultBuffer beginning with a DxfCode.AppName value, and assign it to the entity’s XData property. When reading, check the app name and then iterate the remaining TypedValues. For more complex data, use extension dictionaries instead.

## Database events

> Source: [dotnet/fundamentals/database-events.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/database-events.md)

AutoCAD fires database events whenever objects are added, modified or removed from a drawing. Handling these events allows your plug‑in to react to changes in real time—for example by updating a user interface, logging modifications or enforcing drawing standards. This document explains the major database events, how to subscribe and unsubscribe, and best practices for writing event handlers.

## Key database events

Each Database instance exposes events through its ObjectAppended, ObjectModified, ObjectErased, ObjectReappended and ObjectUnappended events. The event handlers receive an ObjectEventArgs or ObjectErasedEventArgs containing the DBObject that triggered the event. A summary of the common events:

| Event | When it fires | EventArgs type |
| --- | --- | --- |
| ObjectAppended | After a new object is added to the database (e.g. appended to a block table record). | ObjectEventArgs |
| ObjectModified | After an object’s data is modified. | ObjectEventArgs |
| ObjectErased | When an object is erased or unerased. Check the Erased property of the args to see which. | ObjectErasedEventArgs |
| ObjectReappended | When an object is re‑appended to the database (usually after being removed temporarily). | ObjectEventArgs |
| ObjectUnappended | When an object is removed from its owner, but not erased (e.g. moved between owners). | ObjectEventArgs |

There are additional events (ObjectOpenedForModify, ObjectClosed, etc.) but the above are the most commonly used.

## Subscribing to events

Subscribe to database events in your plug‑in’s initialization routine (for example, IExtensionApplication.Initialize()) or when your command is invoked. Always store a reference to the event handlers so you can detach them during cleanup.

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;
using Autodesk.AutoCAD.EditorInput;
using Autodesk.AutoCAD.Runtime;

public class EventDemo : IExtensionApplication
{
    public void Initialize()
    {
        var db = Application.DocumentManager.MdiActiveDocument.Database;
        // subscribe to events
        db.ObjectAppended += Db_ObjectAppended;
        db.ObjectModified += Db_ObjectModified;
        db.ObjectErased   += Db_ObjectErased;
    }

    public void Terminate()
    {
        var db = Application.DocumentManager.MdiActiveDocument.Database;
        // unsubscribe to avoid dangling handlers
        db.ObjectAppended -= Db_ObjectAppended;
        db.ObjectModified -= Db_ObjectModified;
        db.ObjectErased   -= Db_ObjectErased;
    }

    private void Db_ObjectAppended(object sender, ObjectEventArgs e)
    {
        var ent = e.DBObject as Entity;
        if (ent != null)
        {
            Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nAdded: {ent.GetType().Name}, id={ent.ObjectId}");
        }
    }

    private void Db_ObjectModified(object sender, ObjectEventArgs e)
    {
        var ent = e.DBObject as Entity;
        if (ent != null)
        {
            Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nModified: {ent.GetType().Name}, id={ent.ObjectId}");
        }
    }

    private void Db_ObjectErased(object sender, ObjectErasedEventArgs e)
    {
        var ent = e.DBObject as Entity;
        if (ent != null)
        {
            string action = e.Erased ? "Erased" : "Unerased";
            Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\n{action}: {ent.GetType().Name}, id={ent.ObjectId}");
        }
    }
}
```

This example writes messages to the command line when objects are added, modified or erased. Your handlers can perform more complex logic such as updating a palette or logging changes to an external file. Always avoid heavy computations or long‑running tasks inside an event handler—if necessary, queue work to be processed later.

## Database events best practices

- **Do not modify the database from within an event handler.** Modifying objects while handling ObjectModified or ObjectAppended can lead to reentrant events and unpredictable behaviour. If you need to react to a change by modifying another object, schedule the work after the current command completes.
- **Always unsubscribe events when your plug‑in is unloaded.** Failing to detach handlers can cause memory leaks and multiple event firings. Use your Terminate() method or dispose pattern to remove handlers.
- **Check the Erased flag in ObjectErasedEventArgs** to distinguish between erase and unerase operations.
- **Minimise UI interaction in event handlers.** Writing to the command line is generally safe, but opening dialogs or manipulating the ribbon from within events can interfere with AutoCAD’s message loop.

These guidelines follow Autodesk’s recommendations on event handling and are reinforced by the training labs, which use events to populate a tree view and log changes. Adhering to them prevents common pitfalls such as reentrancy and performance issues.

## Database events summary

Database events provide hooks into the lifecycle of objects within AutoCAD drawings. Subscribe to ObjectAppended, ObjectModified, ObjectErased and related events to monitor changes, but remember to keep handlers lightweight, unsubscribe properly and avoid modifying the database from within event callbacks.

## Document events

> Source: [dotnet/fundamentals/document-events.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/fundamentals/document-events.md)

In addition to object‑level database events, AutoCAD notifies you when documents are created, activated or destroyed. Document events are useful for initialising per‑document state, cleaning up resources and responding to changes in the active drawing. This guide explains the key events exposed by DocumentCollection and demonstrates how to subscribe to them.

## Available events

The Autodesk.AutoCAD.ApplicationServices.DocumentCollection class (accessible via Application.DocumentManager) raises several events:

| Event | When it fires | EventArgs type |
| --- | --- | --- |
| DocumentCreated | After a new drawing is created (e.g. NEW command). | DocumentCollectionEventArgs |
| DocumentToBeDestroyed | Immediately before a drawing is closed. | DocumentCollectionEventArgs |
| DocumentDestroyed | After a drawing has been closed and removed from the collection. | DocumentCollectionEventArgs |
| DocumentActivated | When a document becomes the active document. | DocumentCollectionEventArgs |
| DocumentLockModeChanged | When the document’s lock mode changes (read‑only, write, etc.). | DocumentLockModeChangedEventArgs |

These events give you the ability to perform startup and cleanup tasks at appropriate times.

## Subscribing to document events

Register document event handlers during your plug‑in’s initialization. Store the handlers so you can detach them when the plug‑in unloads. The following example logs messages when documents are created, activated and destroyed:

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.Runtime;

public class DocumentEventDemo : IExtensionApplication
{
    public void Initialize()
    {
        var docs = Application.DocumentManager;
        docs.DocumentCreated      += Docs_DocumentCreated;
        docs.DocumentActivated    += Docs_DocumentActivated;
        docs.DocumentToBeDestroyed += Docs_DocumentToBeDestroyed;
        docs.DocumentDestroyed    += Docs_DocumentDestroyed;
    }

    public void Terminate()
    {
        var docs = Application.DocumentManager;
        docs.DocumentCreated      -= Docs_DocumentCreated;
        docs.DocumentActivated    -= Docs_DocumentActivated;
        docs.DocumentToBeDestroyed -= Docs_DocumentToBeDestroyed;
        docs.DocumentDestroyed    -= Docs_DocumentDestroyed;
    }

    private void Docs_DocumentCreated(object sender, DocumentCollectionEventArgs e)
    {
        Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nDocument created: {e.Document.Name}");
        // initialise per‑document data structures here
    }

    private void Docs_DocumentActivated(object sender, DocumentCollectionEventArgs e)
    {
        Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nActivated: {e.Document.Name}");
        // update UI or data structures for the active document
    }

    private void Docs_DocumentToBeDestroyed(object sender, DocumentCollectionEventArgs e)
    {
        Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nClosing: {e.Document.Name}");
        // save state or prompt user before the document closes
    }

    private void Docs_DocumentDestroyed(object sender, DocumentCollectionEventArgs e)
    {
        Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nClosed: {e.Document.Name}");
        // clean up per‑document data structures
    }
}
```

The `DocumentLockModeChanged` event provides information about changes in document lock status. Use it to coordinate multi‑document workflows or to enforce editing restrictions.

## Document events best practices

- **Avoid long operations in event handlers.** Just like database event handlers, document handlers should be quick. Defer heavy processing until after the document is fully loaded or closed.
- **Check for null active document.** In some cases—such as during application shutdown—the active document may be null. Guard against null references when writing to the editor.
- **Use `DocumentLock` when accessing document data from non‑command contexts.** If you need to modify a document as a result of an event, wrap your code in a `using (DocumentLock dl = doc.LockDocument())` block.

## Document events summary

Document events allow you to monitor when drawings are created, activated and destroyed. By subscribing to DocumentCollection events, you can initialise and dispose of per‑document resources and keep your plug‑in’s state in sync with the active drawing. Always unsubscribe event handlers when your plug‑in unloads and keep handlers lightweight.

## Palette sets

> Source: [dotnet/patterns/palette-sets.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/patterns/palette-sets.md)

AutoCAD’s palette sets provide dockable containers for custom user interfaces built with Windows Forms or WPF. They allow your plug‑in to expose complex UI elements, such as forms, tree views or property grids, within the AutoCAD environment. This guide describes how to create, display and manage palette sets using the AutoCAD .NET API.

## Creating a palette set

A palette set is represented by the Autodesk.AutoCAD.Windows.PaletteSet class. You typically create one palette set per plug‑in and add one or more user controls to it. A unique GUID identifies the palette set for persistence across sessions. Example:

```csharp
using Autodesk.AutoCAD.Windows;
using Autodesk.AutoCAD.Runtime;

public class PaletteDemo : IExtensionApplication
{
    private PaletteSet _paletteSet;
    private MyUserControl _control;

    public void Initialize()
    {
        // Create the palette set lazily when the command is executed
    }

    public void Terminate() { }

    [CommandMethod("ShowPalette")]
    public void ShowPalette()
    {
        if (_paletteSet == null)
        {
            // create a new palette set with a title and a GUID
            _paletteSet = new PaletteSet("My Tools", new Guid("A0B1C2D3-E4F5-6789-ABCD-0123456789AB"));
            // instantiate your WinForms or WPF user control
            _control = new MyUserControl();
            // add the control to the palette set; the first parameter is the tab name
            _paletteSet.Add("Main", _control);
            // optional: set initial size and dock state
            _paletteSet.Size = new System.Drawing.Size(300, 400);
            _paletteSet.DockEnabled = DockSides.Left | DockSides.Right;
        }
        // show the palette set
        _paletteSet.Visible = true;
    }
}
```

When the user executes the ShowPalette command, the palette is created if necessary and made visible. Subsequent calls simply show the existing palette. AutoCAD remembers the palette’s position and size between sessions using the GUID.

## Handling palette events

Palette sets raise events such as StateChanged (when the palette is docked, floating or hidden) and StyleChanged (when the appearance changes). You can subscribe to these events to persist state or enable/disable commands. For example:

```csharp
_paletteSet.StateChanged += (s, e) =>
{
    Application.DocumentManager.MdiActiveDocument.Editor.WriteMessage($"\nPalette state: {_paletteSet.Dock}");
};
```

## Using multiple palettes

A single palette set can host multiple tabs. Call Add(tabName, control) for each user control you want to display. You can also create multiple palette sets, but be mindful of cluttering the UI. In Lab 4 and Lab 7 of the training labs, a palette set is used to display a tree view of entities and respond to database events.

## Palette sets best practices

- **Create palettes lazily.** Avoid instantiating heavy controls in your plug‑in’s Initialize method. Create them when the user first runs the command that needs the UI.
- **Use unique GUIDs.** The palette set’s GUID determines how AutoCAD saves its position and size. Changing the GUID resets the palette’s stored state.
- **Lock the document for database writes.** If your palette invokes commands or modifies the database in response to button clicks or selection changes, wrap those operations in a DocumentLock (using (doc.LockDocument()) { … }).
- **Dispose of controls appropriately.** AutoCAD does not automatically dispose your user controls when the palette closes. Implement IDisposable if your control holds unmanaged resources and dispose of it when the palette set is destroyed.

## Palette sets summary

Palette sets enable rich dockable UIs in AutoCAD. Create a PaletteSet with a unique GUID, add user controls, and toggle its visibility from a command. Handle palette events to manage state and wrap database modifications in a DocumentLock when responding to UI interactions.

## Ribbon UI

> Source: [dotnet/patterns/ribbon.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/patterns/ribbon.md)

The AutoCAD ribbon provides a modern command interface that you can extend from .NET. You can add tabs, panels and buttons to expose your commands, and integrate custom options panels into the Options dialog. This page summarises the key APIs and shows patterns drawn from the official training labs.

## Basic ribbon structure

AutoCAD exposes the ribbon through the RibbonControl, which hosts RibbonTabs. Each tab contains one or more RibbonPanels, and panels contain controls such as RibbonButton, RibbonCommandButton, split buttons and combo boxes. To add your own UI elements:

1. Retrieve the top‑level `RibbonControl` via `Autodesk.Windows.ComponentManager.Ribbon` (requires the `Autodesk.Windows` assembly).
2. Create a `RibbonTab` and set its `Title` and `Id`.
3. Create one or more `RibbonPanelSource` objects. Each `RibbonPanelSource` holds a `RibbonPanel`.
4. Create `RibbonButton`s or `RibbonCommandButton`s and assign command handler IDs (`CommandHandler` or `CommandParameter`).
5. Add panels to the tab and add the tab to the ribbon.

```csharp
using Autodesk.Windows;

public void AddRibbon()
{
    RibbonControl ribbon = ComponentManager.Ribbon;
    // create a new tab
    RibbonTab tab = new RibbonTab
    {
        Title = "My Tools",
        Id = "MY_TAB"
    };
    ribbon.Tabs.Add(tab);
    // create a panel
    RibbonPanelSource panelSource = new RibbonPanelSource { Title = "My Panel" };
    RibbonPanel panel = new RibbonPanel { Source = panelSource };
    tab.Panels.Add(panel);
    // create a button that runs a command named MYCOMMAND
    RibbonButton btn = new RibbonButton
    {
        Text = "Do Something",
        ShowText = true,
        ShowImage = false,
        CommandHandler = new RelayCommand(() => AcadApp.DocumentManager.MdiActiveDocument.SendStringToExecute("MYCOMMAND ", true, false, false))
    };
    panelSource.Items.Add(btn);
}
```

In this example, RelayCommand is a simple implementation of ICommand that invokes a delegate. You can also derive from RibbonCommandButton and set its CommandParameter to the name of a command method. For persistent UI, place your ribbon creation code in the IExtensionApplication.Initialize() method.

## Loading ribbon from XML

For more complex layouts or to reuse definitions across plug‑ins, define your ribbon in an XML file. The Autodesk.Windows.RibbonServices class provides a LoadRibbon() method that accepts an XML string. The schema mirrors the ribbon object model. For example:

```xml
<ribbon>
  <tab id="MY_TAB" title="My Tools">
    <panel id="MY_PANEL" title="My Panel">
      <button id="BTN1" text="Do Something" command="MYCOMMAND" />
    </panel>
  </tab>
</ribbon>
```

Load this definition at startup:

```csharp
using Autodesk.Windows;
// assume xmlContent contains the ribbon XML
RibbonServices.LoadRibbon(xmlContent);
```

Buttons defined in XML can specify the command to run via the command attribute. Images can be embedded or referenced by resource name. XML makes it easier to manage large ribbon definitions.

## Context menus and drag‑and‑drop

The training lab also demonstrates adding a custom context menu to the drawing canvas. Use Autodesk.AutoCAD.Windows.ContextMenuExtension to create a menu, add MenuItems and register a click handler. Register the menu using Application.AddDefaultContextMenuExtension() and remove it in your Terminate() method. Always lock the document before modifying the database in a context‑menu callback by calling Document.LockDocument().

Drag‑and‑drop support allows users to drag items from your palette or ribbon into the drawing. Handle the drop event and call your command accordingly. In the labs this is used to run a circle jig when the user drags a label onto the canvas.

## Options dialog tabs

To extend the Options dialog, subscribe to the Application.DisplayingOptionDialog event. When triggered, call e.AddTab() to insert your own tab. Provide a TabbedDialogExtension that hosts a WinForms control and optional OnOK/OnCancel callbacks. Example:

```csharp
public static void AddOptionsTab()
{
    Autodesk.AutoCAD.ApplicationServices.Application.DisplayingOptionDialog += (s, e) =>
    {
        var ctrl = new MyUserControl();
        var ext = new TabbedDialogExtension(ctrl, null, null);
        e.AddTab("My Settings", ext);
    };
}
```

The user control can expose properties that your plug‑in reads when the user presses OK. The training lab uses this mechanism to capture a custom variable from the Options dialog and display it when commands run.

## Ribbon UI summary

Ribbon customisation allows you to integrate your commands into the AutoCAD UI seamlessly. Create tabs, panels and buttons programmatically via Autodesk.Windows types or load them from XML. Add context menus for right‑click actions and custom tabs in the Options dialog. Always perform UI initialisation in the IExtensionApplication.Initialize() method and clean up in Terminate().

## Options dialog integration

> Source: [dotnet/patterns/options-dialog.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/patterns/options-dialog.md)

AutoCAD’s Options dialog can host custom tabs provided by plug‑ins. Adding a tab allows you to expose configuration settings to users in a familiar place. The AutoCAD .NET API exposes this capability via the Autodesk.AutoCAD.ApplicationServices.TabbedDialogExtension class and the Application.DisplayingOptionDialog event. This guide shows how to create and manage a custom options tab.

## Basic pattern

1. Implement `IExtensionApplication` and subscribe to `Application.DisplayingOptionDialog` in your `Initialize()` method.
2. In the event handler, call `e.AddTab(title, extension)` to add your custom tab. The `TabbedDialogExtension` constructor takes your user control and optional callbacks for OK, Cancel and Help.
3. Read values from your control in the OK callback and persist them as appropriate.
4. Unsubscribe from the event in your `Terminate()` method.

Example:

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.Runtime;

public class OptionsDemo : IExtensionApplication
{
    public static string MySetting = string.Empty;

    public void Initialize()
    {
        // register handler when the options dialog is displayed
        Application.DisplayingOptionDialog += OnDisplayingOptions;
    }

    public void Terminate()
    {
        Application.DisplayingOptionDialog -= OnDisplayingOptions;
    }

    private void OnDisplayingOptions(object sender, TabbedDialogEventArgs e)
    {
        // create your user control
        var ctrl = new MyOptionsControl();
        // define OK and Cancel callbacks
        System.Action ok = () => { MySetting = ctrl.TextBoxValue; };
        System.Action cancel = () => { /* revert changes if necessary */ };
        // wrap the control in a tabbed dialog extension
        var ext = new TabbedDialogExtension(ctrl, ok, cancel);
        // add a new tab with title "My Settings"
        e.AddTab("My Settings", ext);
    }
}
```

Your `MyOptionsControl` can be a Windows Forms or WPF user control containing text boxes, check boxes, etc. Use the OK callback to read values from the control and store them in static variables or write them to configuration files. If you need to validate input, perform validation before closing the dialog.

## Reading and applying settings

To apply settings when your commands run, read the static property or configuration you updated in the OK callback. For example, in a command method:

```csharp
[CommandMethod("UseSetting")]
public static void UseSetting()
{
    var ed = Application.DocumentManager.MdiActiveDocument.Editor;
    ed.WriteMessage($"\nCurrent value of MySetting: {OptionsDemo.MySetting}");
    // use the setting to control command behaviour
}
```

## Multiple tabs

You can add multiple tabs by calling e.AddTab() more than once. Each tab can have its own control and callbacks. Be mindful that adding too many tabs may clutter the Options dialog.

## Options dialog best practices

- **Keep controls lightweight.** Avoid heavy operations or database access in the options controls. Use the controls solely to gather input.
- **Validate input.** Use error providers or message boxes to inform users of invalid input when they click OK.
- **Unregister handlers.** Always remove your `DisplayingOptionDialog` handler in `Terminate()` to prevent memory leaks.

## Options dialog summary

Custom tabs in the Options dialog provide a convenient way to expose settings. Register the DisplayingOptionDialog event, create a TabbedDialogExtension with your user control and callbacks, and add the tab via AddTab(). Persist settings in the OK callback and retrieve them in your commands.

## Electrical engineering patterns

> Source: [dotnet/patterns/electrical-engineering.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/patterns/electrical-engineering.md)

This page collects patterns specific to substation protection & control (P&C) and broader electrical design workflows. Engineers at Root 3 Power use these techniques every day to automate drawing registers, locate device symbols, parse layer names and manage external references across dozens of drawings. The patterns here build on the core transaction and attribute concepts documented elsewhere in this repository and assume readers are already comfortable with the basic transaction model.

## Reading title‑block fields

Electrical drawings typically include a title block with attributes such as project number (PROJ_NO), drawing number (DWG_NO), revision (REV), date (DATE) and engineer initials (ENGINEER). These attributes live on the block reference that represents the title block. To read them:

1. Open the drawing and start a transaction using the pattern from the transaction model guide.
2. Iterate over model‑space entities and locate the title‑block by its block name (for example `TBLOCK` or company‑specific names). For dynamic blocks the block definition may have an anonymous name starting with `*U`; get the effective name via `bref.DynamicBlockTableRecord`.
3. For each `BlockReference`, iterate its `AttributeCollection` and match attribute tags.
4. Read the `TextString` for each tag and store the values.

```csharp
using Autodesk.AutoCAD.ApplicationServices;
using Autodesk.AutoCAD.DatabaseServices;
using Autodesk.AutoCAD.Runtime;

/// <summary>
/// Reads common title‑block fields from the active drawing.
/// </summary>
[CommandMethod("ReadTitleBlock")] // command name: READTITLEBLOCK
public static void ReadTitleBlock()
{
    var doc = Application.DocumentManager.MdiActiveDocument;
    var db  = doc.Database;
    var ed  = doc.Editor;
    using (var tr = db.TransactionManager.StartTransaction())
    {
        // open the block table and modelspace
        var bt = (BlockTable)tr.GetObject(db.BlockTableId, OpenMode.ForRead);
        var ms = (BlockTableRecord)tr.GetObject(bt[BlockTableRecord.ModelSpace], OpenMode.ForRead);

        foreach (ObjectId id in ms)
        {
            if (!id.ObjectClass.IsDerivedFrom(RXObject.GetClass(typeof(BlockReference))))
                continue;
            var bref = (BlockReference)tr.GetObject(id, OpenMode.ForRead);

            // Effective block name – handles dynamic block references
            string name = bref.Name;
            if (name.StartsWith("*U"))
            {
                var dynDef = (BlockTableRecord)tr.GetObject(bref.DynamicBlockTableRecord, OpenMode.ForRead);
                name = dynDef.Name;
            }
            if (!name.Equals("TITLEBLOCK", StringComparison.OrdinalIgnoreCase))
                continue;

            // Gather attribute values
            string projNo = null, dwgNo = null, rev = null, date = null, engineer = null;
            foreach (ObjectId attId in bref.AttributeCollection)
            {
                var att = (AttributeReference)tr.GetObject(attId, OpenMode.ForRead);
                switch (att.Tag.ToUpperInvariant())
                {
                    case "PROJ_NO": projNo   = att.TextString; break;
                    case "DWG_NO":  dwgNo    = att.TextString; break;
                    case "REV":     rev      = att.TextString; break;
                    case "DATE":    date     = att.TextString; break;
                    case "ENGINEER":engineer = att.TextString; break;
                }
            }
            ed.WriteMessage($"\nProject={projNo}, Drawing={dwgNo}, Rev={rev}, Date={date}, Engineer={engineer}");
        }
        tr.Commit();
    }
}
```

Use this pattern whenever you need to extract document metadata or populate a drawing register. Always open the BlockReference for read and only upgrade individual attributes to ForWrite when modifying them.

## Batch scanning a project folder

Root 3 Power projects often contain hundreds of DWG files that must be catalogued into a drawing register. Use a headless database and the batch processing pattern to loop through files without opening AutoCAD’s UI. For each DWG:

1. Create a new `Database` with `new Database(false, true)`.
2. Call `db.ReadDwgFile(path, FileOpenMode.OpenForReadAndAllShare, false, null)` to load the drawing.
3. Start a transaction and call the `ReadTitleBlock` routine above to extract attributes.
4. Store the extracted metadata in a CSV or data structure.
5. Dispose the database before moving to the next file.

```csharp
foreach (string file in Directory.EnumerateFiles(projectFolder, "*.dwg"))
{
    using (var db = new Database(false, true))
    {
        db.ReadDwgFile(file, FileOpenMode.OpenForReadAndAllShare, false, null);
        using (var tr = db.TransactionManager.StartTransaction())
        {
            // reuse the title‑block reading logic
            // (could be factored into a helper method as shown above)
            // write results to a CSV or list
            tr.Commit();
        }
    }
}
```

This pattern allows you to build a drawing register that lists project and sheet numbers, revisions and engineer initials without manually opening each file.

## Finding relay, switch and device symbols

Electrical schematics rely on block definitions to represent relays, switches, transformers and other devices. To find all instances of a particular symbol across a drawing set, use selection filters with wild‑cards. AutoCAD’s selection filters support wild‑card characters such as * (match any sequence) and ? (match a single character). The following example selects all INSERT entities whose block name matches a particular prefix:

```csharp
using Autodesk.AutoCAD.EditorInput;
// find all relay blocks whose names start with "RELAY_"
TypedValue[] filter = new TypedValue[]
{
    new TypedValue((int)DxfCode.Start, "INSERT"),
    // filter on block name using wild‑card
    new TypedValue((int)DxfCode.BlockName, "RELAY_*")
};
var selFilter = new SelectionFilter(filter);
var result = Application.DocumentManager.MdiActiveDocument.Editor.GetSelection(selFilter);
if (result.Status == PromptStatus.OK)
{
    foreach (SelectedObject sel in result.Value)
    {
        // process each relay instance
    }
}
```

When scanning multiple files you can combine this pattern with the headless batch pattern to count or list devices across the entire project. If the devices are dynamic blocks, check the effective name via DynamicBlockTableRecord before comparison.

## Parsing layer naming conventions

Root 3 Power uses structured layer names to encode discipline, function and status. A common convention is DISC-FUNC-STATUS, for example:

| Sample layer | Discipline | Function | Status |
| --- | --- | --- | --- |
| P-CTL-EXIST | P (protection) | CTL (control wiring) | EXIST (existing) |
| C-WIR-NEW | C (control) | WIR wiring | NEW (new work) |

You can parse these names by splitting on the hyphen and mapping each segment to a domain meaning. A helper method might look like this:

```csharp
public (string discipline, string function, string status) ParseLayerName(string name)
{
    var parts = name.Split('-');
    return (parts.Length > 0 ? parts[0] : string.Empty,
            parts.Length > 1 ? parts[1] : string.Empty,
            parts.Length > 2 ? parts[2] : string.Empty);
}
```

After parsing you can enforce standards (e.g. ensure discipline is one of P, C, E, etc.) or automatically assign objects to layers based on function.

## Working with external references (XREFs)

Drawings often reference other drawings (xrefs) such as symbol libraries or other sheets. Managing these references is important when packaging deliverables. Each block definition has properties that indicate whether it comes from an external reference (BlockTableRecord.IsFromExternalReference) and the path of the referenced file (PathName). To list all xrefs and convert their paths to relative:

```csharp
using Autodesk.AutoCAD.DatabaseServices;
using System.IO;

// iterate block table records and handle external references
foreach (ObjectId id in bt)
{
    var btr = (BlockTableRecord)tr.GetObject(id, OpenMode.ForRead);
    if (btr.IsFromExternalReference)
    {
        ed.WriteMessage($"\nXRef: {btr.Name} -> {btr.PathName}");
        // change absolute path to relative: open the record for write and compute a relative path
        btr.UpgradeOpen();
        btr.PathName = Path.GetRelativePath(projectRoot, btr.PathName);
    }
}
```

This code opens each xref block record for write using UpgradeOpen() and uses Path.GetRelativePath() from System.IO to compute a relative path. There is no Upgrader class or RelativePathTo extension in the AutoCAD API; both were references to a DevBlog helper and should not be used in production. When you need to embed an xref permanently, call the Bind method on the database: binding an xref makes it a permanent part of the drawing
help.autodesk.com
. Conversely, you can remove an xref definition altogether using DetachXref. Always check that the referenced file is not locked before binding or detaching.

## Batch find and replace

Replacing text across many drawings requires selection filters and wildcard patterns. AutoCAD’s wild‑card characters include * for any sequence and ? for any single character. The following pattern selects all MText objects containing a string and replaces it:

```csharp
TypedValue[] tv = new TypedValue[2];
tv[0] = new TypedValue((int)DxfCode.Start, "MTEXT");
tv[1] = new TypedValue((int)DxfCode.Text, "*OLDVALUE*");
var filter = new SelectionFilter(tv);
var res = ed.GetSelection(filter);
if (res.Status == PromptStatus.OK)
{
    using (var tr = db.TransactionManager.StartTransaction())
    {
        foreach (SelectedObject sel in res.Value)
        {
            var mtext = (MText)tr.GetObject(sel.ObjectId, OpenMode.ForWrite);
            mtext.Contents = mtext.Contents.Replace("OLDVALUE", "NEWVALUE");
        }
        tr.Commit();
    }
}
```

To replace text in `DBText` or AttributeReference objects, adjust the filter’s Start code to "TEXT" or "INSERT" and upgrade each object before modifying its TextString. Combine this with the batch scanning pattern to perform find/replace across an entire project folder.

Caveat: filtering MText entities by DxfCode.Text uses the object's raw contents, which may include RTF‑like formatting codes. A wild‑card filter can miss matches if the text contains formatting. For reliable results iterate all MText entities and inspect the Contents or plain Text property directly instead of relying solely on the selection filter.

## AutoCAD .NET gotchas table

> Source: [dotnet/gotchas.md](https://github.com/chamber-19/autocad-knowledge/blob/main/dotnet/gotchas.md)

This document collects known failure modes encountered when developing AutoCAD .NET plugins. Refer to this table before debugging your own code. Each symptom is accompanied by its cause and a concise fix. Only evidence‑based issues from documentation or observed behaviour are included.

| Symptom | Cause | Fix |
| --- | --- | --- |
| Plugin crashes AutoCAD on load | Autodesk assemblies are copied local to the plugin, causing version conflicts | Set `<Private>False</Private>` on all Autodesk references in your .csproj so they are not copied into the plugin output |
| eNotOpenForWrite on property set | Object opened ForRead and then modified | Call entity.UpgradeOpen() before any property assignment |
| eWasErased exception | The object was erased in the current or a previous transaction | Check DBObject.IsErased before accessing or skip erased objects |
| Command registered but does nothing | Missing or incorrect [assembly: CommandClass] attribute | Ensure the assembly‑level CommandClass attribute references the class containing your commands |
| Attributes not visible after block insert | The AttributeReference was not added or not registered | Use BlockReference.AppendAttribute() (or assign attributes in a loop) and call tr.AddNewlyCreatedDBObject() for each new attribute |
| Dynamic block shows *U5 as name | Dynamic block references point at anonymous block definitions | Use BlockReference.DynamicBlockTableRecord to obtain the effective name |
| ReadDwgFile throws sharing violation | The DWG file is open or locked by another process | Use FileOpenMode.OpenForReadAndAllShare to allow shared reads or ensure the file is closed |
| Changes not saved after Commit() in batch processing | Batch edits were made in a temporary Database but SaveAs was not called | After committing, call db.SaveAs(path, DwgVersion.Current) to persist changes |
| Plugin loaded but commands are unrecognised | The project targets the wrong .NET framework for the AutoCAD version | Target .NET 8 for AutoCAD 2026 or .NET 4.8 for older releases |
| COM call deadlocks AutoCAD | Calling the COM automation API from inside the AutoCAD process blocks the message loop | Move COM calls to an external process (e.g., a Python sidecar) and avoid COM within managed plugins |
| MText or DBText does not update visually | The entity's graphics cache is not marked dirty after editing its text | After setting TextString, call RecordGraphicsModified(true) |
| Plugin works in 2024 but crashes in 2026 | Assembly version mismatch between plugin and AutoCAD installation | Reference the exact versions of acdbmgd.dll, acmgd.dll and accoremgd.dll from the target AutoCAD installation |
| New entities disappear after commit | AddNewlyCreatedDBObject() was not called for the new object | Always call AddNewlyCreatedDBObject(newObj, true) after appending any entity |
| Event handler executes multiple times | Event handlers were not unsubscribed | Remove event handlers in Terminate() or when they are no longer needed using -= |
| Lock violation when creating an extension dictionary | Attempting to write to an extension dictionary opened ForRead | Call UpgradeOpen() on the dictionary before adding items |
| Title block fields out of sync | Attributes updated by index instead of tag; dynamic blocks not handled | Search attributes by tag and handle dynamic block names using DynamicBlockTableRecord |
| Transaction aborted unexpectedly | Exceptions thrown inside the transaction without catching | Wrap transaction logic in try/catch/finally and call Commit() only when no exceptions occur |

Use this table as a checklist when a plugin misbehaves. Fixes should be verified against the official documentation or working training code.

## Pattern: block count by name

> Source: [patterns/block-count-by-name.md](https://github.com/chamber-19/autocad-knowledge/blob/main/patterns/block-count-by-name.md)

**Verified:** 2026-04-25 on NANULAK 180 MW BESS substation drawing
**Use case:** Count instances of a specific block to verify quantity, audit
placement, or build a count-based BOM line item

### The prompt

```text
Count all instances of block <BLOCK_NAME>
```

Replace `<BLOCK_NAME>` with the exact block name (e.g., `CATL TOP`, `PANEL-100A`).

### What Assistant does internally

1. `discoverAutoCADTypes` — looks up `blockreference` schema
2. `queryAutoCADObjects` with filter `name = '<BLOCK_NAME>'` — returns all
   block reference instances with positions and layer info
3. Counts results, reports total
4. **Bonus behavior:** Assistant analyzes spatial arrangement and infers
   patterns when present (e.g., "arranged in a 3×18 grid pattern")

### Verified result

- Input: query for "CATL TOP" blocks
- Output: 54 instances on EQUIP layer, arranged in 3 columns × 18 rows
- Assistant correctly inferred this was a battery storage array for the
  180 MW BESS facility — domain reasoning, not just data extraction

### Edge cases and gotchas

- **Block name must match exactly.** Case sensitivity varies; if a query
  returns 0, try variations of the name. Block names in R3P drawings
  sometimes have spaces, sometimes have hyphens; check the actual block
  name with `INSERT` or by clicking an existing instance.
- **Counts block REFERENCES, not block DEFINITIONS.** If "CATL TOP" appears
  54 times in the drawing, you get 54 — even though there's only 1 block
  definition. This is what you want for BOM purposes.
- **Cannot filter by attribute values.** If you need "all panels with
  voltage = 480V", this pattern won't help. Use R3P's batch FNR tool
  instead (see `limitations/block-attributes-not-queryable.md`).

### When this is the right pattern

- BOM line item: "how many of X do we have on this drawing"
- Layout audit: confirm placement matches design intent
- Cross-checking against schedule documents
- Any count-based takeoff (light fixtures, panels, terminal blocks, etc.)

### When to use something else

- Need to filter or count by attribute values: use batch FNR
- Need to count blocks across multiple drawings: this is single-drawing only
- Need to see WHICH blocks (not just count): add "show me" or "list" to the
  prompt to get position/layer info

### Related patterns

- [polyline-length-by-layer.md](polyline-length-by-layer.md) — for linework
  takeoffs
- [closed-polyline-area-by-layer.md](closed-polyline-area-by-layer.md) — for
  area takeoffs

### Bonus: spatial pattern inference

Assistant noticed the 3×18 grid arrangement of CATL TOP blocks unprompted and
correctly identified it as a BESS battery array. This kind of interpretive
layer is genuinely useful for sanity-checking — if Assistant says "I see 36
units in a 6×6 grid" and your design called for 54 in 3×18, that's a
discrepancy worth investigating.

For drawings where layout matters (BESS arrays, parking lots, panel banks,
foundations), include "describe the arrangement" in your prompt to invoke
this behavior explicitly.

## Pattern: closed polyline area by layer

> Source: [patterns/closed-polyline-area-by-layer.md](https://github.com/chamber-19/autocad-knowledge/blob/main/patterns/closed-polyline-area-by-layer.md)

**Verified:** 2026-04-25 on NANULAK 180 MW BESS substation drawing
**Use case:** Area takeoff for surface materials, foundations, fenced zones,
gravel areas, or any closed polyline outline that represents a quantity-of-area

```text
Sum the area of all closed polylines on layer <LAYER_NAME>
```

Replace `<LAYER_NAME>` with the relevant layer (e.g., `Foundation`,
`GRAVEL_SURFACE`, `FENCE_AREA`).

1. `discoverAutoCADTypes` — looks up polyline schema, identifies `area` as
   a queryable property on closed polylines
2. `queryAutoCADObjects` with filter for closed polylines on specified layer
3. Sums areas, reports total in drawing units squared
4. **Bonus behavior:** Carries unit context from earlier session queries.
   If you previously confirmed drawings are in inches, Assistant
   automatically converts to square feet and acres without being asked.
5. **Bonus behavior:** Buckets results by area size when patterns emerge
   (large/medium/small/pad)

- Input: 18 closed polylines on Foundation layer
- Output: 1,406,736 sq drawing units → 9,769 sq ft → 0.224 acres
- Assistant grouped foundations by approximate size:
  - Large (~918 sq ft each): 3 units
  - Medium (~800 sq ft each): 3 units
  - Small (~197 sq ft each): 3 units
  - Several smaller pad foundations

- **"Closed" means the polyline's endpoints connect.** If a polyline is
  drawn as a rectangle outline but the last segment doesn't snap closed,
  it has no `area` property and won't be counted. This is a drafting
  quality issue, not a pattern issue. Sanity-check: if your foundation
  count seems low, check for unclosed polylines.
- **Drawing units squared, not raw units.** Assistant returns the area in
  the unit² of the drawing. For inches → sq inches → sq feet (÷144) → sq
  acres (÷43,560). Verify the unit chain.
- **Doesn't differentiate hatched vs unhatched.** If your foundation layer
  contains only the outlines and the hatching is on a separate layer, this
  works. If the layer contains both outlines and hatch boundaries, you
  might double-count. Layer hygiene matters.

- Foundation takeoff for concrete volume estimation (multiply by depth)
- Gravel/asphalt surfacing area for ordering
- Fenced or restricted zone area for permitting documentation
- Any "how much area does X cover" review task

- Need volume directly: this only gives area; you multiply for volume
- Need to differentiate by foundation type beyond size buckets: use
  attributes if available, or filter by layer naming conventions
- Need to compare planned vs as-built areas: this is single-snapshot

- [polyline-length-by-layer.md](polyline-length-by-layer.md) — for linework
  perimeter measurements
- [block-count-by-name.md](block-count-by-name.md) — when foundations are
  block references rather than polyline outlines

### Note on the size-bucket interpretation

Assistant grouping the 18 foundations into large/medium/small/pad was useful
but unprompted. If you want the raw individual areas, ask: "list the
individual areas without grouping" — otherwise Assistant defaults to the
interpreted summary.

## Pattern: polyline length by layer

> Source: [patterns/polyline-length-by-layer.md](https://github.com/chamber-19/autocad-knowledge/blob/main/patterns/polyline-length-by-layer.md)

**Verified:** 2026-04-25 on NANULAK 180 MW BESS substation drawing
**Use case:** Material takeoff for linework that represents quantity-of-stuff
(grounding rebar, fence, conduit, pipe runs, cable trays)

```text
Sum the total length of all polylines on layer <LAYER_NAME> and report it
in feet. I need to order <MATERIAL>.
```

Replace `<LAYER_NAME>` with the actual layer name (e.g., `4-O BARE GROUND - BESS`)

1. `discoverAutoCADTypes` — looks up the schema for polyline types
2. `queryAutoCADObjects` with filter `layer.name = '<LAYER_NAME>'` — gets
   matching polylines with their `length` property
3. If results exceed pagination cap (~200), additional `queryAutoCADObjects`
   calls retrieve remaining items
4. Sums client-side, returns total in drawing units, prompts about unit
   conversion

- Input: 209 polylines on layer "4-O BARE GROUND - BESS"
- Output: 233,847.7 drawing units → 19,487.3 feet (after unit clarification)
- Assistant correctly identified this as a substantial-but-reasonable amount
  for a 180 MW BESS facility grounding grid

- **Drawing units are ambiguous.** Assistant returns "drawing units" by default
  and offers conversion options (inches, mm). For substations, drawings are
  almost always in inches. Always sanity-check: would 233,847 feet of rebar
  fit in a substation footprint? Obviously not, so it's inches → feet (÷12).
- **Pagination is automatic.** You don't need to manage it manually. Assistant
  detects when the result count exceeds what was returned and fetches the rest.
- **Layer name must match exactly.** The filter syntax is case-insensitive but
  the layer name string must match. If you're not sure, ask: "What layers in
  this drawing contain polylines?" first.

- One-off material takeoff during design review
- Periodic BOM checks against current drawing state
- Quick sanity check on whether a quantity changed after revisions

- Need to add waste/connection allowance: do that math yourself or in a
  spreadsheet (don't ask Assistant — it'll guess at percentages)
- Need to break down by sub-area or zone: filter by additional criteria like
  `layer.name = 'X' AND <region condition>` or use multiple specific layers
- Need to track changes over time: this is a snapshot tool, not a versioned
  takeoff system

### The prompts

For a basic list:

```text
List all text on layer <LAYER_NAME>
```

1. `discoverAutoCADTypes` — looks up `text` and `mtext` schemas
2. `queryAutoCADObjects` for both `text` and `mtext` types on the specified

   retrieved together
4. Returns enumerated list or formatted table

## Gotcha: AutoCAD busy state

> Source: [gotchas/autocad-busy-state.md](https://github.com/chamber-19/autocad-knowledge/blob/main/gotchas/autocad-busy-state.md)

**Discovered:** 2026-04-25 during text enumeration test

### What happens

You send a prompt to Assistant. It tries to execute, possibly retries with
different approaches, then returns a message like:

> "I'm unable to query the drawing at the moment because AutoCAD is busy
> with another operation. This could be due to:
>
> - A document operation in progress (opening, creating, or closing a document)
> - Another command or process running in AutoCAD
> - A dialog box that needs attention"

### Why it happens

The MCP server runs in-process inside AutoCAD. When you're mid-command
(drawing a line that hasn't been completed, in the middle of an editor
prompt, with a modal dialog open, etc.), the AutoCAD command queue is
blocked from accepting external requests.

Assistant's MCP tool calls go through that same queue, so they queue up
behind your active operation. After a timeout, Assistant gives up and
reports the busy state.

### How to avoid

**Before running Assistant queries**, ensure AutoCAD is in idle state:

- No active commands (press ESC a couple of times if unsure)
- No open dialogs (close any property panels, modify dialogs, etc.)
- No drawing/closing/saving operations in progress
- Mouse not actively dragging or hovering on a tool tip

The command line should show the bare prompt with no current command.

### How to recover

1. Look at AutoCAD's status — what command/operation is active
2. Cancel or complete it (ESC for command cancel, OK/Cancel on dialogs)
3. Confirm the command line is clean
4. Tell Assistant "try again" or restate the original prompt

This was verified during testing: a query that returned busy errors
succeeded immediately after completing the active operation in AutoCAD
and re-prompting.

### Workflow implications

For real engineering workflows, this means:

- **Don't have AutoCAD and Assistant fighting for the same drawing.** If
  you're actively drafting, finish your current command before asking
  Assistant a question.
- **Long-running plugin operations will block Assistant.** If you're
  running a custom R3P plugin that takes 30 seconds, Assistant queries
  during that window will fail.
- **Background operations matter.** Save operations, regen, file I/O can
  all temporarily make AutoCAD "busy."

For automation pipelines, this is a real constraint to design around.
You can't just fire Assistant queries blindly during a workflow — you
need explicit synchronization with AutoCAD's idle state.

### Future investigation

It would be useful to know if there's a way to detect "AutoCAD is idle"
programmatically before sending queries, rather than retrying on busy
errors. Filed in `future-work/` as a research item.

## Gotcha: automatic pagination

> Source: [gotchas/pagination-is-automatic.md](https://github.com/chamber-19/autocad-knowledge/blob/main/gotchas/pagination-is-automatic.md)

**Discovered:** 2026-04-25 during rebar takeoff (209 polylines) and text
table enumeration (479 mtext entities)

You ask Assistant for "all polylines on layer X" or similar broad query.
Assistant runs `queryAutoCADObjects` and the response includes a hint that
there are more results than what was returned (typically capped around 200).

Assistant automatically issues a second query with `skip` set to retrieve
the remaining items, then combines results before reporting.

You see this in Assistant's status messages:

> ✓ Executing queryAutoCADObjects…
> I found polylines on the layer, but there are more results than what was
> returned (209 total, 200 returned). Let me get the remaining polylines:
> ✓ Executing queryAutoCADObjects…

### Why it’s worth knowing

It's not a bug — it's working correctly. But:

1. **It takes longer.** Two MCP tool calls instead of one. Don't be
   surprised by the apparent stall between the first "executing" message
   and the final result.

2. **It uses more of your context window** (if you're using Assistant
   alternatives via API). The full result set goes through Assistant's
   reasoning, so very large queries (thousands of items) might bump
   against context limits.

3. **You can short-circuit it.** If you only need a sample to confirm
   the data is queryable, ask for a limited result first:
   > "Get the first 10 polylines on layer X with their lengths"
   This is faster and uses less context. Use the full query only when
   you actually need to sum/aggregate everything.

### When to think about it

- **Drawings with 500+ entities of the type you're querying.** Expect
  multiple round-trips.
- **Drawings with 5,000+ entities.** Might hit Assistant's tolerance for
  long responses; consider breaking into multiple narrower queries.
- **You only need a count, not the data.** Ask "how many" rather than
  "list all" — Assistant will use the metadata count rather than
  enumerating.

### How to leverage it deliberately

For very large drawings, structure prompts in stages:

1. First: "How many polylines on layer X are there?" → get count
2. Second (only if count is reasonable): "Sum their lengths" → get total
3. Third (only if you need the breakdown): "List individual lengths" →
   get the data

This avoids paying the cost of full enumeration when you only need a
summary number.

## Gotcha: unit ambiguity

> Source: [gotchas/unit-ambiguity.md](https://github.com/chamber-19/autocad-knowledge/blob/main/gotchas/unit-ambiguity.md)

**Discovered:** 2026-04-25 during rebar takeoff testing

You ask Assistant for a length, area, or other measurement. Assistant returns
a number followed by "drawing units" (or sq drawing units for areas).

The drawing units could be inches, feet, millimeters, meters, or anything
else. AutoCAD itself doesn't always have a definitive answer — drawings can
be set up with any unit convention and there's no universal "I am inches"
metadata that's always present.

If you don't catch this, you can order materials based on the wrong number.

### Real example

During the rebar takeoff test:

- Assistant returned: "Total length in feet: 233,847.7 feet"
- The drawing was actually in inches, so that number was sq inches
- Correct value: 19,487.3 feet

The Bessfacility footprint isn't 233,847 feet across (that's 44 miles).
The magnitude check immediately caught the error.

Assistant's first instinct on a "report in feet" prompt is to assume the
number is already feet (since you said feet). It does the conversion only
if you flag it. The MCP server returns raw numbers without unit metadata
attached, so there's nothing to disambiguate from.

1. **Tell Assistant the drawing units upfront.** First message of the
   session, before any queries:
   > "I'm working on R3P substation drawings. Drawings are in inches by
   > default unless I tell you otherwise."

2. **Always sanity-check magnitude.** Ask yourself: does this number make
   physical sense given the size of the facility? A 180 MW BESS substation
   is roughly 5-10 acres. 21,000 ft of grounding rebar is plausible.
   233,000 ft (44 miles) is not.

3. **Ask Assistant explicitly to convert.** If you say "report in feet"
   and the result looks ten times too big, ask: "Was that already in feet,
   or are those inches that need to be divided by 12?"

### How to recover if you hit it

Assistant carries unit context within a session. Once you correct the
unit assumption (e.g., "Probably inches because this is a substation and
233,847.7 feet is a lot"), Assistant will:

- Acknowledge the correction
- Recalculate with the right conversion
- Apply the corrected unit assumption to subsequent queries in the same
  session

This was tested and verified: after the inch correction during the rebar
takeoff, the next query (foundation areas) automatically used inches as
the assumption without re-prompting.

### Bigger lesson

Engineering judgment is irreplaceable. Assistant correctly executes the
math; you have to know whether the answer is plausible. The pairing works
because each side covers the other's blind spots — Assistant doesn't tire
when iterating over 209 polylines, you don't accept 233,000 ft of rebar
when 19,487 is the truth.

## Gotcha: SDK conflicts with Autodesk plugins

> Source: [gotchas/sdk-conflict-with-autodesk-plugins.md](https://github.com/chamber-19/autocad-knowledge/blob/main/gotchas/sdk-conflict-with-autodesk-plugins.md)

When a chamber-19 AutoCAD plugin ships its own copy of a NuGet package that is also shipped by an Autodesk first-party plugin, the .NET CLR loads only one version into `acad.exe`. If versions differ, plugin startup can fail with `TypeLoadException` or `MissingMethodException` at runtime.

### Symptom

Plugin fails to initialize. Observed examples:

- `System.TypeLoadException: Could not load type ''X'' from assembly ''Y, Version=A.B.C.D''` where the type exists in the version your plugin references but not in the version actually loaded.
- `System.MissingMethodException` for methods present in your version but absent in the loaded version.

### Root cause

The .NET CLR resolves simple-named assembly references on a first-wins basis within the same `AssemblyLoadContext`. AutoCAD plugins all load into the default ALC. When two plugins reference the same package at different versions, only one DLL ends up loaded — typically whichever plugin was loaded first into the process.

### Concrete example (May 2026)

Investigation during chamber-19-autocad-mcp commit 2 development surfaced this conflict:

| DLL | Loaded from | Source |
| --- | --- | --- |
| `ModelContextProtocol.dll` | `C:\Program Files\Autodesk\ApplicationPlugins\AutoCAD-MCP-Server-2027.bundle\Contents\` | Autodesk `acmcp` plugin (v0.4.0.0 — built from a `0.4.0-preview.x` release) |
| `ModelContextProtocol.Core.dll` | same path | Autodesk `acmcp` plugin (v0.4.0.0) |
| `ModelContextProtocol.AspNetCore.dll` | chamber-19-autocad-mcp build output | chamber-19 plugin (v1.3.0) |

chamber-19's plugin referenced `ModelContextProtocol.AspNetCore` 1.3.0, which depends on `ModelContextProtocol.Core` 1.3.0''s type `ISseEventStreamStore`. But `Core` 0.4.0.0 was already loaded from Autodesk''s bundle, and `ISseEventStreamStore` does not exist in 0.4.0.0 → `TypeLoadException` at startup.

### Diagnostic procedure

When a plugin fails with type-load errors involving a third-party package, run this from PowerShell while AutoCAD is open with the failing plugin loaded (or attempting to load):

```powershell
Get-Process acad | Select-Object -First 1 -ExpandProperty Modules |
  Where-Object { $_.ModuleName -match '' } |
  Select-Object ModuleName, FileName, FileVersion,
    @{n='ProductVersion';e={$_.FileVersionInfo.ProductVersion}}
```

This reveals (a) which DLLs of that package family are loaded, (b) which paths they came from, and (c) the version actually present in the process. If multiple paths show up, you have a conflict. If a single path shows up that is not yours, your plugin''s reference has been silently overridden by another plugin''s copy.

### Remediation patterns

Three patterns are viable. All have real trade-offs; pick deliberately.

### Pattern 1 — Pin to the conflicting version

Downgrade your plugin''s reference to match the version already loaded in the process. Coexistence becomes automatic.

- **Pros:** Zero loader engineering, no new code patterns.
- **Cons:** Lose access to API surface the newer version added. If the conflicting plugin pins a preview version, you ship against a preview SDK. Future Autodesk version bumps can break you.
- **Use when:** the older version''s API is sufficient and you can tolerate Autodesk''s release cadence dictating yours.

### Pattern 2 — `AssemblyLoadContext` isolation

Wrap the component that depends on the conflicting package in a custom `AssemblyLoadContext` that loads its own copy, with selective default-fallback for shared types (e.g. `Autodesk.AutoCAD.*` types must come from the host ALC, not the isolated one).

- **Pros:** Architecturally clean. Survives Autodesk version bumps. No dependency on what other plugins load.
- **Cons:** New pattern in the codebase. ALC debugging surface is non-trivial. Requires splitting your plugin into a thin shell (default ALC) plus a host (custom ALC).
- **Use when:** you intend to distribute the plugin to machines where Autodesk''s bundle may be present.

### Pattern 3 — Disable the conflicting plugin

Rename the conflicting bundle directory (e.g. `AutoCAD-MCP-Server-2027.bundle` → `AutoCAD-MCP-Server-2027.bundle-disabled`) so AutoCAD doesn''t load it. Document the rename as an installation precondition for your plugin.

- **Pros:** Zero engineering. Reversible (rename, not delete).
- **Cons:** Coexistence broken. If a user re-enables the bundle or installs both plugins on a fresh machine, your plugin fails until the rename is restored. Not a viable distribution stance.
- **Use when:** dev-box-only territory; you need to verify your design works before investing in Pattern 2.

### Related gotcha — `LoadOnRequest="true"` does not always prevent loading

Autodesk''s `acmcp` plugin''s `PackageContents.xml` declares `LoadOnRequest="true"`, which is documented to mean the plugin loads only when its commands are invoked. However, in current AutoCAD 2027 builds (verified May 2026 via the diagnostic procedure above), the plugin''s DLLs were observed in `acad.exe`''s loaded modules at startup — before any user command was issued.

Possible explanations:

- AutoCAD 2027''s plugin loader changed `LoadOnRequest` semantics in a recent build.
- Another Autodesk component (Assistant panel, ProductIcon registration, automatic registration scan) loads the bundle eagerly regardless of the manifest setting.
- Earlier autocad-knowledge notes about `LoadOnRequest` were incomplete.

Until this is confirmed, **do not rely on `LoadOnRequest="true"` to prevent another plugin''s assemblies from being loaded into your process.** The diagnostic procedure is the only reliable way to know what is actually loaded.

## Limitation: block attributes not queryable

> Source: [limitations/block-attributes-not-queryable.md](https://github.com/chamber-19/autocad-knowledge/blob/main/limitations/block-attributes-not-queryable.md)

**Discovered:** 2026-04-25
**Affected workflows:** Anything that depends on block attribute values
(panel schedules, equipment tagging, status fields like ISSUED FOR APPROVAL)

### What we tried

```text
Find all blocks with attribute DESC2=ISSUED FOR APPROVAL
```

### What Assistant said

> "I cannot check for block attributes like 'DESC2' because **the system
> doesn't support querying block attributes**. The available blockreference
> schema only includes basic block properties but not attribute data."
>
> Available block properties Assistant can query:
>
> - Block name
> - Position coordinates
> - Rotation angle
> - Scale factors (X, Y, Z)
> - Layer information
>
> What it cannot access:
>
> - Block attributes (like DESC2, DESC1, etc.)
> - Attribute values
> - Attribute properties

### Why this matters

Block attributes are how a lot of R3P drawings carry structured data:

- Panel schedules (panel number, voltage, amperage, location)
- Equipment tagging (equipment ID, manufacturer, model, serial)
- Status fields (drawing state, revision, designer)
- Title block data (project name, drawing number, scale, date)

If Assistant can't read these, it can't answer questions like:

- "How many panels are tagged 'IFP' but should be 'IFC'?"
- "List all equipment with manufacturer = 'Siemens'"
- "Find all blocks where the revision attribute is empty"

## Workaround: R3P's batch FNR tool

R3P has a Batch Find & Replace tool that operates at the attribute level —
it can read and write block attribute values across drawings at scale.
For attribute-driven workflows, batch FNR is the right tool.

The split is now:

- **Geometry queries** → Assistant
- **Attribute queries and bulk attribute updates** → batch FNR

### Hybrid possibility (not yet implemented)

Conceptually, Assistant could be the front-end that DECIDES which attributes
to change (via natural-language reasoning), and batch FNR could be the
back-end that EXECUTES the change at scale. Same pattern as the rebar
takeoff: LLM reasoning + tool execution.

This would require integration work between Assistant prompts and batch FNR
that doesn't exist today. Filed in `future-work/` as a possibility.

### When this might change

If Autodesk extends the MCP server schema in a future AutoCAD release to
include block attributes, this limitation goes away. Worth re-testing with
each major AutoCAD release. For now (AutoCAD 2027 as of April 2026), the
limitation is structural.

## Limitation: plain polyline is not a wire

> Source: [limitations/plain-polyline-not-wire.md](https://github.com/chamber-19/autocad-knowledge/blob/main/limitations/plain-polyline-not-wire.md)

**Discovered:** 2026-04-25
**Affected workflows:** Wire tracing, terminal connection analysis, signal
flow review, anything that depends on knowing "this wire connects A to B"

On a wiring schematic drawing where wires were drawn as plain polylines:

```text
Find all wires under Left side view on right door that connect to actual
Cable tags
```

> "Based on the search results, I can see that AutoCAD Electrical has
> specific tools for tracing wire connections and viewing cable information,
> but I cannot directly query the specific electrical schematic elements
> you're asking about using the current AutoCAD MCP tools available to me."
>
> "I cannot directly find and analyze wires connected to cable tags in
> AutoCAD Electrical schematics because **the current system doesn't support
> querying AutoCAD Electrical-specific objects** like:
>
> - Wire networks and connections
> - Cable tags and their associated wires
> - Electrical component attributes and connections
> - Door panel views and their specific wire routing"

### The deeper issue

The wires in this drawing weren't ACADE wire entities — they were plain
polylines drawn on a wires layer. From AutoCAD's perspective:

- A polyline has length, color, layer, start/end points
- A polyline does NOT have a wire number, source terminal, destination
  terminal, voltage, or any other electrical metadata
- Connectivity is visual, not semantic — two polylines that happen to touch
  at endpoints aren't a "connection" in any data sense

ACADE (AutoCAD Electrical) wire objects are different:

- They ARE proper wire entities with all the metadata above
- They participate in wire networks with traced connectivity
- They're queryable as "wires" with cable tag associations
- ACADE has commands like Wire From/To reports, trace tools, etc.

This is **the architectural fork** for electrical work going forward.

If you draft electrical schematics in vanilla AutoCAD with plain polylines,
agent automation hits a wall the moment you need anything beyond raw
geometry. Wire numbering, cable tagging, connectivity tracing, terminal
mapping — none of it is available.

If you draft in ACADE with proper wire entities, all of that data becomes
queryable in principle. (Whether the MCP server exposes ACADE-specific
queries is a separate question — Assistant indicated it currently doesn't,
even when run inside ACADE. This may improve in future versions.)

### Workaround for current state

For wire-tracing and terminal-connection work right now:

1. **Use ACADE's built-in tools** (Trace environment, Wire From/To reports,
   Connections pane, Cable From/To reports). These work from ACADE's UI,
   not via Assistant, but they DO understand the connectivity.
2. **Build R3P plugins that read ACADE's data directly** via the .NET API.
   The data is there in the drawing database; ACADE-aware plugins can
   reach it even when Assistant can't.
3. **For brand new electrical work, draft in ACADE.** This costs more
   upfront (different tool, different conventions) but pays off in
   automation capability later.

### Recommended R3P-internal decision

Going forward, electrical work that's likely to need agent automation
(wire tracing, BOM generation from schematics, automated cable schedules)
should be drafted in ACADE rather than vanilla AutoCAD.

Vanilla AutoCAD is fine for:

- Site plans and civil drawings
- Mechanical layouts
- One-off conceptual drawings
- Existing-condition documentation

ACADE should be the default for:

- Schematic diagrams
- Wiring diagrams
- Panel layouts with wire interconnections
- Anything that will need wire/terminal queries

This is a non-trivial workflow decision and should be made at the project
level, not retroactively imposed on existing drawings.

## Limitation: MCP SDK conflict with Autodesk bundle

> Source: [limitations/mcp-sdk-conflict-with-autodesk-bundle.md](https://github.com/chamber-19/autocad-knowledge/blob/main/limitations/mcp-sdk-conflict-with-autodesk-bundle.md)

**Discovered:** 2026-05-08
**Affected workflows:** Installing `chamber19-autocad-mcp` into an AutoCAD 2027 environment that still has Autodesk's `AutoCAD-MCP-Server-2027.bundle` enabled

Install and load `chamber19-autocad-mcp` while Autodesk's shipped MCP bundle remains enabled in `ApplicationPlugins`.

Both plugins ship the `ModelContextProtocol` SDK and both load into AutoCAD's default `AssemblyLoadContext`. Autodesk pins an older MCP SDK version, while `chamber19-autocad-mcp` uses a newer one. In that mixed state, the Chamber 19 plugin fails at startup with a `TypeLoadException`, `MCPSTATUS` reports that the MCP server failed to start, and the HTTP listener never binds. [chamber-19-autocad-mcp/README.md†L7-L17]

This is not a transient startup glitch. It is a structural loader conflict between two in-process bundles that both expect to own the same dependency identity. Until the load boundary changes, "install both and hope AutoCAD picks the right one" is not a safe deployment model. [chamber-19-autocad-mcp/README.md†L7-L17]

### Workaround

Disable Autodesk's bundle by renaming its folder before installing the Chamber 19 plugin:

```powershell
Rename-Item "C:\Program Files\Autodesk\ApplicationPlugins\AutoCAD-MCP-Server-2027.bundle" `
            "AutoCAD-MCP-Server-2027.bundle-disabled"
```

Restart AutoCAD after the rename. This is the current documented workaround in the plugin README. [chamber-19-autocad-mcp/README.md†L9-L17]

### Long-term fix

The durable fix is to isolate the MCP SDK in its own `AssemblyLoadContext` so the Chamber 19 plugin can load side-by-side with Autodesk's bundle without sharing the default context. That direction is already called out in the plugin's issue tracker and README; until it lands, treat coexistence as unsupported. [chamber-19-autocad-mcp/README.md†L9-L17]
